# SKILL 06c: 规则构建（规划）

> 定义基于本体的推理规则，将历史学领域知识形式化为可执行的推理规则。

---

## 一、功能概述

本体定义了类、属性和关系的结构，但复杂的领域知识需要通过规则来表达。本功能计划构建推理规则系统，实现：

- **领域规则**：历史学和史记特有的知识规则
- **推理规则**：从已知事实推导新知识
- **验证规则**：检查数据的合理性和一致性

---

## 二、核心功能

### 2.1 领域规则

**目的**：将历史学和史记领域知识形式化

**规则类型**：

**时代规则**：
- 如果人物在秦统一（前221年）之前活动，则标记为"先秦人物"
- 如果人物在汉高祖即位（前202年）后活动，则标记为"汉代人物"
- 如果事件发生在楚汉之际（前206-前202），则标记为"楚汉争霸"相关事件

**身份规则**：
- 如果人物在本纪中，则身份为"帝王"
- 如果人物在世家中，且有封国，则身份为"诸侯"
- 如果人物在列传中，且标题为"X列传"，则身份为"X"（如刺客、游侠）

**关系推理规则**：
- 如果A是B的父亲，B是C的父亲，则A是C的祖父
- 如果A是B的君主，C是B的部下，且C侍奉A，则C也是A的部下
- 如果A和B是兄弟，B和C是兄弟，则A和C是兄弟（传递性）

**地理规则**：
- 如果地点A属于郡X，郡X属于国Y，则地点A也属于国Y
- 如果事件发生在地点A，地点A属于国X，则事件发生在国X

**应用场景**：
- 自动标注人物时代和身份
- 补全缺失的关系
- 推断事件的地理归属

### 2.2 推理规则

**目的**：从已知事实推导新知识

**推理类型**：

**生卒年推断**：
- 如果人物在某年任官，则该年人物活着
- 如果人物A是人物B的父亲，且B生于X年，则A可能生于X-20~X-50年
- 如果人物参与某事件，则在事件发生时活着

**关系推断**：
- 如果A和B共同侍奉君主C，则A和B是同僚
- 如果A攻打B，且A胜利，则A和B是对手关系
- 如果A封B为某官，则A和B是君臣关系

**事件关系推断**：
- 如果事件A是事件B的原因，事件B是事件C的原因，则A间接导致C
- 如果事件A和B同时发生，地点相邻，人物重叠，则可能是同一事件

**应用场景**：
- 补全知识图谱的空缺
- 发现隐含的关系
- 丰富实体属性

### 2.3 验证规则

**目的**：检查数据的合理性

**验证类型**：

**时间一致性**：
- 人物生年不能晚于卒年
- 事件发生时间不能早于参与者的生年
- 君臣关系必须在双方生存期间重叠

**关系一致性**：
- 父子关系不能成环
- 一个人不能同时是另一个人的父亲和儿子
- 对手关系应该是对称的

**逻辑一致性**：
- 人物不能同时在不同地点
- 事件的参与者必须在该时期活着
- 官职的任命者必须有权力任命该官职

**应用场景**：
- 数据导入时验证
- 定期数据质量检查
- 发现标注错误

## 三、技术基础

### 3.1 规则语言

计划使用的规则表达方式：

- **SWRL（Semantic Web Rule Language）**：标准的语义Web规则语言
  - 与OWL本体集成
  - 支持推理引擎执行
- **SPARQL Rules**：基于SPARQL的规则定义
- **Python规则引擎**：灵活的自定义规则实现

### 3.2 数据准备

本功能依赖已完成的工作：

- **本体模型**：类、属性、关系定义（参考 [SKILL_06a](SKILL_06a_实体到本体.md)、[SKILL_06b](SKILL_06b_属性关系约束构建.md)）
- **实体数据**：已标注的实体和关系（参考 [SKILL_03](SKILL_03_实体构建.md)、[SKILL_05](SKILL_05_关系构建.md)）
- **领域知识**：历史学专家经验和史记研究成果

- 从历史学文献提取领域规则

- 规则性能优化
- 冲突规则检测和解决
- 规则版本管理

## 七、规则示例

以下为规则的概念性示例（非实际代码）：

### 示例1：祖父关系推理
```
IF 人物A 是 人物B 的父亲
AND 人物B 是 人物C 的父亲
THEN 人物A 是 人物C 的祖父
```

### 示例2：先秦人物标记
```
IF 人物X 活动年代 < 前221年
THEN 人物X 属于类别 "先秦人物"
```

### 示例3：时间一致性验证
```
IF 人物A 生年 > 人物A 卒年
THEN 报错 "生卒年矛盾"
```
