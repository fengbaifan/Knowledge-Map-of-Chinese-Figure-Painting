# SKILL 01g: 标注符号集合原则

> **核心原则**：标点符号、标注符号、Markdown符号三大集合完全分离（disjoint），互不重叠，确保文本解析的确定性和可维护性。

---

## 一、问题诊断

### 1.1 为什么需要Disjoint原则

**符号冲突的危害**：
1. **解析歧义**：同一符号在不同上下文有不同含义，导致解析器混淆
2. **渲染错误**：标注符号被误认为Markdown语法，导致显示异常
3. **维护困难**：人工阅读时难以区分符号的作用域
4. **脚本脆弱**：正则表达式需要大量转义和特殊处理

**示例（符号冲突）**：
```markdown
❌ 问题1：标点与Markdown冲突
《史记》曰："天下大势，分久必合。"
（引号" "如果用ASCII双引号，会与Markdown代码块冲突）

❌ 问题2：标注与标点冲突
〖@项羽〗曰："吾闻〖@汉王〗。"
（引号与标注符号混用，难以区分层次）

❌ 问题3：标注与Markdown冲突
〖$三万人〗出*兵*
（$与Markdown数学公式冲突，*与强调符号冲突）
```

### 1.2 Disjoint原则的优势

✅ **解析确定性**：每个符号有唯一的语义，解析器行为可预测
✅ **渲染稳定性**：不会误触Markdown/HTML渲染规则
✅ **维护清晰性**：人类和机器都能准确识别符号作用
✅ **脚本简洁性**：正则表达式简单，无需复杂转义

## 二、三大符号集合定义

### 2.1 标点符号集合（Punctuation Set）

**定义**：用于断句、语气、引用的符号，属于原文的一部分。

**完整列表**（13类）：

| 符号 | Unicode | 名称 | 用途 |
|------|---------|------|------|
| 。 | U+3002 | 全角句号 | 句子结束 |
| ，| U+FF0C | 全角逗号 | 句内停顿 |
| 、 | U+3001 | 顿号 | 并列词语 |
| ；| U+FF1B | 全角分号 | 复句分隔 |
| ：| U+FF1A | 全角冒号 | 提示总结 |
| ？| U+FF1F | 全角问号 | 疑问句 |
| ！| U+FF01 | 全角叹号 | 感叹句 |
| " | U+201C | 左双引号 | 第一层引用开始 |
| " | U+201D | 右双引号 | 第一层引用结束 |
| ' | U+2018 | 左单引号 | 第二层引用开始 |
| ' | U+2019 | 右单引号 | 第二层引用结束 |
| 《 | U+300A | 左书名号 | 书名开始 |
| 》 | U+300B | 右书名号 | 书名结束 |
| —— | U+2014×2 | 破折号 | 注释转折 |
| …… | U+2026×2 | 省略号 | 文字省略 |

**规范**：
- ✅ 只使用全角标点（CJK Punctuation范围）
- ❌ 禁止半角标点：`,` `.` `;` `:` `?` `!` `"` `'`
- ❌ 禁止ASCII引号：`"` `'` `<` `>`
- ❌ 禁止直角引号：「」『』（日文用法）

### 2.2 标注符号集合（Annotation Set）

**定义**：用于语义标记的符号，不属于原文，用于知识抽取。

#### 2.2.1 实体标注符号（〖TYPE〗格式）

**外层括号**：
- 左括号：`〖`（U+3016，白色六角括号左）
- 右括号：`〗`（U+3017，白色六角括号右）

**类型标记符号**（18种）：

> **📌 邦国符号变更历史**：
> - **v1.0-v2.x** (2026-03-31前)：使用 `'` (U+0027, 单引号) 标注邦国
> - **v3.0** (2026-04-01起)：迁移至 `◆` (U+25C6, 黑菱形)
> - **迁移原因**：单引号与标点符号冲突，且不符合Disjoint原则
> - **⚠️ 已知问题**：`◆` 与动词标注 `◈` 视觉相似，建议未来进一步优化
> - **详细记录**：见 [SKILL_03a1_标注符号迁移.md](references/SKILL_03a1_标注符号迁移.md)

| 符号 | 类型 | 说明 | Markdown冲突 |
|------|------|------|-------------|
| `@` | 人名 | 历史人物 | - |
| `=` | 地名 | 城邑山川 | - |
| `;` | 官职 | 正式职衔 | - |
| `#` | 身份 | 社会角色 | ⚠️ 标题符号 |
| `%` | 时间 | 年号纪年 | - |
| `&` | 氏族 | 宗族家族 | - |
| `◆` | 邦国 | 政治实体 | ⚠️ 与动词`◈`视觉相似 |
| `^` | 名物 | 典章名物（含礼制/律历/医学等专门术语） | - |
| `~` | 族群 | 民族部落 | ⚠️ 删除线（某些扩展） |
| `•` | 器物 | 礼器兵器 | - |
| `!` | 天文 | 星象历法 | - |
| `?` | 神话 | 传说神话 | - |
| `+` | 生物 | 动植物 | - |
| `{` | 典籍 | 古籍书名 | ⚠️ 属性语法（某些扩展） |
| `:` | 礼仪 | 礼仪祭祀 | ⚠️ 自定义块 `:::` |
| `[` | 刑法 | 刑罚法律 | ⚠️ 链接/段落编号 |
| `_` | 思想 | 哲学概念 | ⚠️ 斜体 |
| `$` | 数量 | 计量数值 | ⚠️ 数学公式（某些扩展） |

**⚠️ 与Markdown有交集的符号（需要未来修正）**：
- `#`：Markdown标题符号（`# 标题`），但在 `〖#身份〗` 中被括号包裹，实际无冲突
- `_`：Markdown斜体符号（`_text_`），在 `〖_思想〗` 中无冲突，但不直观
- `[`：Markdown链接和段落编号（`[1]`），在 `〖[刑法〗` 中可能引起混淆
- `$`：Markdown数学公式（`$E=mc^2$`，某些渲染器），在 `〖$数量〗` 中无冲突
- `:`：Markdown自定义块（`:::`），在 `〖:礼仪〗` 中无冲突
- `{`：Markdown属性语法（`{.class}`，某些扩展），在 `〖{典籍〗` 中无冲突
- `~`：Markdown删除线（`~~text~~`，某些扩展），在 `〖~族群〗` 中无冲突

**当前安全性评估**：
- ✅ **短期可用**：因为类型标记在 `〖〗` 内部，Markdown解析器不会识别
- ⚠️ **长期建议**：未来版本应替换有冲突的符号为更独特的Unicode字符
- 🔄 **优先级**：低（当前设计可用，但影响可读性和直观性）

**消歧分隔符**：
- `|`（U+007C，竖线）：用于`〖TYPE 显示名|规范名〗`
- ⚠️ 与Markdown表格符号冲突，但在 `〖〗` 内部无实际问题

#### 2.2.2 动词标注符号（⟦TYPE⟧格式）

**外层括号**：
- 左括号：`⟦`（U+27E6，数学左白方括号）
- 右括号：`⟧`（U+27E7，数学右白方括号）

**类型标记符号**（4种）：

| 符号 | 类型 | 说明 | Unicode | 视觉特征 |
|------|------|------|---------|---------|
| `◈` | 军事 | 攻伐征战 | U+25C8 | 白色菱形含黑点 |
| `◉` | 政治 | 立废诛封 | U+25C9 | 鱼眼圆形 |
| `○` | 外交 | 使朝聘盟 | U+25CB | 白圆 |
| `◇` | 经济 | 赐赋税贸 | U+25C7 | 白菱形 |

**⚠️ 视觉混淆警告**：
- **问题符号对**：`〖◆周〗`（实体-邦国）vs `⟦◈攻〗`（动词-军事）
- **混淆原因**：两者都是菱形系列，`◆`（黑菱形）和`◈`（白菱形含点）在快速阅读时难以区分
- **影响范围**：邦国实体（周、齐、秦等）经常与军事动词（攻、伐等）共现，如"〖◆秦〗⟦◈攻〗〖◆赵〗"
- **改进建议**：见 §2.2.3 实体-动词符号协调设计

#### 2.2.3 实体-动词符号协调设计原则 ⚠️

**核心问题**：实体标注和动词标注的TYPE符号视觉相似，导致快速阅读时混淆。

**设计原则**（未来符号选择时必须遵守）：

1. **形状系列分离原则**
   - ✅ 实体符号使用一个形状系列（如：圆形系）
   - ✅ 动词符号使用另一个形状系列（如：方形系）
   - ❌ 禁止两类符号使用相同形状系列（如都用菱形）

2. **视觉区分度测试**
   - 在实际文本中并列出现时，能在0.5秒内区分
   - 测试案例：`〖TYPE1实体〗⟦TYPE2动词〗〖TYPE3实体〗`
   - 通过标准：10人测试，错误率<10%

3. **语义关联优先级**
   - P1：视觉区分度（必须）
   - P2：符号语义关联（建议）
   - P3：输入便捷性（考虑）

**当前问题分析**：

| 维度 | 实体标注 | 动词标注 | 冲突分析 |
|-----|---------|---------|---------|
| **外层括号** | `〖〗` | `⟦⟧` | ✅ 明显不同 |
| **符号系列** | 混合（ASCII+菱形） | 菱形+圆形 | ⚠️ 菱形系冲突 |
| **问题符号** | `◆` (邦国) | `◈` (军事) | ❌ 都是菱形 |
| **共现频率** | 高 | 高 | ⚠️ 经常并列 |

**改进方案（建议）**：

**方案A：替换邦国符号**（推荐，影响范围小）
- 将 `◆` 替换为非菱形系符号
- 候选符号：`⊚` (U+229A, 环形点) 或 `◎` (U+25CE, 靶心圆)
- 优势：只需迁移邦国标注（1种实体类型）
- 劣势：需要再次执行符号迁移

**方案B：重新设计动词符号**（影响大）
- 将菱形系动词符号（`◈◇`）替换为方形系
- 优势：动词符号更统一（全用方形）
- 劣势：需迁移4种动词标注，影响范围大

**方案C：保持现状，加强视觉提示**（临时）
- 在HTML渲染时，实体和动词使用不同底色
- 实体：浅蓝底色，动词：浅黄底色
- 优势：无需迁移，立即生效
- 劣势：纯文本环境下仍然混淆

**推荐实施路径**：
1. **短期**：方案C（HTML渲染优化）
2. **中期**：方案A（邦国符号迁移至 `⊚` 或 `◎`）
3. **长期**：全面审查所有18种实体+4种动词符号，建立完整的视觉区分矩阵

### 2.3 Markdown符号集合（Markdown Set）

**定义**：用于文档结构和格式的符号，属于标记语言语法。

**完整列表**：

## 三、Disjoint原则规范

### 3.1 核心原则

**完全分离原则**：
```
P ∩ A ∩ M = ∅
```
其中：
- P = 标点符号集合（Punctuation）
- A = 标注符号集合（Annotation）
- M = Markdown符号集合（Markdown）

**任意两两不相交**：
```
P ∩ A = ∅  （标点与标注不重叠）
P ∩ M = ∅  （标点与Markdown不重叠）
A ∩ M = ∅  （标注与Markdown不重叠）
```

### 3.2 符号选择策略

#### 策略1：标点符号优先使用Unicode CJK区

**原则**：
- ✅ 使用U+3000-U+303F（CJK Punctuation）
- ✅ 使用U+FF00-U+FFEF（Fullwidth Forms）
- ✅ 使用U+2000-U+206F（General Punctuation，弯引号等）
- ❌ 禁止使用ASCII范围（U+0020-U+007F）

**理由**：
- ASCII符号大量用于Markdown语法（`#` `*` `-` `[` `]` `` ` ``等）
- CJK全角符号不会与Markdown冲突
- 弯引号（" " ' '）是现代中文标点规范用法

#### 策略2：标注符号使用稀有Unicode括号

**原则**：
- ✅ 实体标注使用`〖〗`（U+3016/U+3017，白色六角括号）
- ✅ 动词标注使用`⟦⟧`（U+27E6/U+27E7，数学白方括号）
- ❌ 禁止使用常见括号：`()` `[]` `{}` `（）` `【】`

**理由**：
- `[]` 用于Markdown链接和段落编号
- `{}` 用于某些Markdown扩展语法
- `（）` 可能在原文中出现（注释用法）
- 稀有Unicode括号不会与原文或Markdown冲突

#### 策略3：类型标记符号避开标点和Markdown（⚠️ 当前版本未完全遵守）

**当前状态**：
- ✅ 使用ASCII符号：`@ = ; # % & ' ^ ~ • ! ? + { : [ _ $`
- ✅ 在`〖〗`或`⟦⟧`内部，被括号保护，Markdown解析器不会识别
- ⚠️ **但部分符号与Markdown有交集**：`# _ [ $ : { ~ |`（详见§2.2.1）

**当前可用性**：
- 短期：✅ 可用（括号提供保护，无实际解析冲突）
- 长期：⚠️ 建议修正（影响可读性，不符合严格的Disjoint原则）

**理由（为何当前设计可用但不理想）**：
- ✅ **技术可行**：符号在`〖〗`内部时，Markdown解析器不会识别
- ✅ **输入方便**：ASCII符号易于手工标注
- ⚠️ **可读性差**：`〖#身份〗` `〖_思想〗` `〖[刑法〗` 在阅读时容易与Markdown混淆
- ⚠️ **维护性弱**：未来如果更换渲染器，可能产生新的冲突

**未来改进方向**：
- 用独特的Unicode符号替换有冲突的类型标记
- 优先替换：`# _ [` （高频Markdown符号）
- 低优先级：`$ : { ~ |`（低频或扩展语法）

### 3.3 当前符号冲突分析

#### 冲突1：`$`符号（数量标记 vs Markdown数学公式）

**现状**：
- 标注：`〖$三万人〗`
- Markdown数学：`$E=mc^2$`（某些Markdown渲染器支持）

**分析**：
- ✅ **无冲突**：`$`在`〖〗`内部，Markdown解析器不会识别为数学公式
- ✅ **可区分**：`〖$`是固定模式，`$`单独出现才是数学公式

**结论**：当前设计安全，无需修改。

#### 冲突2：`#`符号（身份标记 vs Markdown标题）

**现状**：
- 标注：`〖#天子〗`
- Markdown标题：`# 标题`

**分析**：
- ✅ **无冲突**：`〖#`是固定模式，在括号内部
- ✅ **可区分**：Markdown标题的`#`在行首，且后面有空格

**结论**：当前设计安全，无需修改。

#### 冲突3：`|`符号（消歧分隔 vs Markdown表格）

**现状**：
- 标注：`〖@台|吕台〗`
- Markdown表格：`| 列1 | 列2 |`

**分析**：
- ✅ **无冲突**：`|`在`〖〗`内部，不在行首
- ✅ **可区分**：Markdown表格的`|`在行首或跨多列

**结论**：当前设计安全，无需修改。

#### 冲突4：`[`符号（刑法标记 vs Markdown链接/段落编号）

**现状**：
- 标注：`〖[族诛〗`
- Markdown段落编号：`[1]` `[1.1]`
- Markdown链接：`[文字](url)`

**分析**：
- ✅ **无冲突**：`〖[`是固定模式，与`[`单独出现不同
- ⚠️ **潜在混淆**：人工阅读时，`[`出现频率高，可能混淆

**结论**：当前设计可用，但需要在lint脚本中明确检测规则。

#### 冲突5：`{`符号（典籍标记 vs Markdown部分扩展）

**现状**：
- 标注：`〖{春秋〗`
- Markdown某些扩展：`{.class}` `{#id}`（属性语法）

**分析**：
- ✅ **无冲突**：`〖{`是固定模式
- ✅ **可区分**：Markdown属性语法的`{`单独出现

**结论**：当前设计安全，无需修改。

### 3.4 符号冲突总结

**当前状态评估**：

| 维度 | 评估 | 说明 |
|-----|------|------|
| **技术可行性** | ✅ 可用 | 括号保护机制有效，无实际解析冲突 |
| **严格Disjoint** | ⚠️ 部分违反 | 8个类型标记与Markdown有交集（`# _ [ $ : { ~ |`） |
| **可读性** | ⚠️ 不理想 | `〖#身份〗` `〖_思想〗` 等容易与Markdown混淆 |
| **维护性** | ⚠️ 有风险 | 未来渲染器变更可能产生新问题 |

**关键设计决策（当前版本）**：
1. ✅ **标点符号全部使用CJK全角区**，避开ASCII → **完全Disjoint**
2. ✅ **标注符号使用稀有Unicode括号** `〖〗` `⟦⟧` → **完全Disjoint**
3. ⚠️ **类型标记符号在括号内部**，依赖括号保护 → **技术可用，但非严格Disjoint**

**需要注意的边界情况**：
- Markdown数学公式（`$...$`）：与`〖$数量〗`在括号外可能冲突
- Markdown表格（`|`）：与`〖TYPE 名|规范名〗`中的消歧符可能混淆
- 段落编号（`[数字]`）：与`〖[刑法〗`中的`[`可能视觉混淆
- Markdown标题（`# 标题`）：与`〖#身份〗`在行首可能混淆

**⚠️ 未来改进计划（低优先级）**：

## 四、符号使用禁忌

### 4.1 禁止混用的符号

| 禁止情况 | 说明 | 正确做法 |
|---------|------|---------|
| ❌ 半角标点 | 引号`"` `'`、逗号`,`、句号`.` | 使用全角：" " ' '，。 |
| ❌ 直角引号 | 「」『』 | 使用弯引号：" " ' ' |
| ❌ 标点在标注内 | `〖@项羽，〗` | 标点在外：`〖@项羽〗，` |
| ❌ Markdown在标注内 | `〖@**项羽**〗` | 标注在外：`**〖@项羽〗**` |
| ❌ 嵌套标注 | `〖#〖@太子〗〗` | 选择一个类型：`〖#太子〗` |
| ❌ 标注跨标点 | `〖@项羽，字〗` | 分开标注：`〖@项羽〗，字` |

### 4.2 符号层次规则

**正确的嵌套层次**（由外到内）：
```
Markdown外层 > 标注符号 > 标点符号 > 原文内容
```

**示例**：
```markdown
✅ 正确层次：
**〖@项羽〗曰："〖@沛公〗安在？"**
└─Markdown┘ └标注┘ └标点┘ └标注┘  └标点┘

❌ 错误层次1：
〖@**项羽**〗
（Markdown在标注内部，违反层次）

❌ 错误层次2：
"〖@项羽曰〗"
（标点包裹标注，应该是标注包裹原文）
```

## 五、符号冲突检测方法

### 5.1 检测逻辑

符号冲突检测需要验证以下6个方面：

#### 检测1：禁止半角标点
- **检查逻辑**：在移除标注符号和段落编号后，查找文本中的半角标点 `,.:;?!"\' `
- **注意**：需要排除标注内部的类型标记符号（如 `〖@name〗` 中的 `@`）
- **排除**：段落编号 `[N.N]` 中的句点、Markdown自定义块 `:::` 中的冒号

#### 检测2：禁止直角引号
- **检查逻辑**：查找日文风格的直角引号 `「」『』`
- **修正建议**：改为弯引号 `" " ' '`

#### 检测3：标点在标注内部
- **检查逻辑**：在 `〖TYPE content〗` 模式中，检查 `content` 是否包含标点符号（除消歧符 `|`）
- **正确做法**：标点应在标注外部，如 `〖@项羽〗，〖@刘邦〗`
- **错误示例**：`〖@项羽，刘邦〗`

#### 检测4：Markdown在标注内部
- **检查逻辑**：在 `〖TYPE content〗` 模式中，检查 `content` 是否包含Markdown符号 `* _ ` [ ] >`
- **层次原则**：Markdown外层 > 标注符号 > 原文内容

#### 检测5：嵌套标注
- **检查逻辑**：检测 `〖TYPE1 ... 〖TYPE2 ... 〗 ... 〗` 模式
- **修正策略**：拍平或选择最合适的类型标记
- **已有工具**：`lint_text_integrity.py --check-nested` 已实现此功能

#### 检测6：标注跨越标点
- **检查逻辑**：在 `〖TYPE content〗` 中，检查 `content` 是否包含句子边界标点 `。！？`
- **错误示例**：`〖@项羽曰。〗` 应改为 `〖@项羽〗曰。`

## 七、FAQ

### Q1: 为什么`$`符号可以用于数量标记，不会与Markdown数学公式冲突？

**A**: 因为`$`在`〖$数量〗`中是固定模式，Markdown解析器会识别：
- `〖$三万人〗` → 整体不是数学公式（`〖`开头）
- `$E=mc^2$` → 明确的数学公式（单独的`$`包裹）

**关键**：`〖$`是固定前缀，与`$`单独出现不同。

### Q2: 标点符号可以在标注内部吗？

**A**: **不可以**。标点是原文的一部分，标注是语义标记：

❌ **错误**：`〖@项羽，字籍〗`
✅ **正确**：`〖@项羽〗，字〖@籍〗`

**原因**：
- 标点属于原文，标注去除后应保留标点
- 标点在内部会导致显示时标点被包裹在标注样式中

### Q3: Markdown粗体可以包裹标注吗？

**A**: **可以**。Markdown在标注外层：

✅ **正确**：`**〖@项羽〗**`（粗体包裹标注）
❌ **错误**：`〖@**项羽**〗`（标注包裹Markdown）

**原则**：Markdown外层 > 标注符号 > 原文内容

### Q4: 段落编号`[1]`会与刑法标记`〖[族诛〗`冲突吗？

**A**: **不会冲突**，但需要明确规范：

- **段落编号**：`[数字]`，单独出现，不参与实体标注
- **刑法标记**：`〖[刑法名〗`，有`〖`前缀

**lint脚本**：可以区分这两种模式。

### Q5: 如何处理原文中已有的括号（如注释）？

**A**: 原文括号保持不变，不与标注符号冲突：

- **原文括号**：`（注释内容）`（全角圆括号）
- **标注括号**：`〖TYPE 内容〗`（白色六角括号）

**关键**：使用不同的Unicode字符，完全disjoint。
