# SKILL 08d: 正确性评估（规划）

> 建立系统的质量评估体系，验证知识准确性，持续保障知识库质量。

---

## 一、功能概述

知识库的价值在于正确性。错误的知识不仅无用，还可能误导用户。本功能计划建立正确性评估体系：

- **准确性验证**：检查知识是否正确
- **一致性检查**：发现矛盾和冲突
- **可信度评估**：评估知识的可靠程度
- **质量监控**：持续跟踪质量指标

---

## 二、正确性维度

### 2.1 事实准确性

**问题**：知识是否符合历史事实？

**验证方法**：
- 与权威史料对比（《史记》原文、《汉书》等）
- 与学术专著核对
- 多源交叉验证
- 专家审查

**错误类型**：
- 实体识别错误（张良误标为李斯）
- 关系标注错误（父子关系标反）
- 时间错误（生卒年错误）
- 事件描述错误（结果标注相反）

### 2.2 逻辑一致性

**问题**：知识内部是否自洽？

**验证方法**：
- 时间一致性（生年<卒年）
- 关系一致性（父子不成环）
- 属性一致性（同一实体的多个属性不矛盾）
- 推理一致性（推理结果与已知事实不冲突）

**矛盾类型**：
- 时间矛盾："A在X年出生，Y年参战"（X>Y）
- 关系矛盾："A是B的祖父，B是A的祖父"
- 空间矛盾："A同时在两地出现"
- 因果矛盾："事件A导致B，B导致A"

### 2.3 完整性

**问题**：知识是否完整无缺失？

**验证方法**：
- 必填字段检查
- 关键关系检查（如人物无父母关系）
- 孤立节点检查
- 覆盖度评估（参考SKILL_08b）

**缺失类型**：
- 属性缺失：人物无生卒年
- 关系缺失：事件无参与者
- 上下文缺失：事件无时间地点
- 链接缺失：实体无原文引用

### 2.4 规范性

**问题**：知识是否符合标注规范？

**验证方法**：
- 格式检查（标注符号、字段格式）
- 命名规范检查
- 分类规范检查
- Schema一致性检查

**违规类型**：
- 格式错误：标注符号错位
- 命名不规范：实体名称不统一
- 分类错误：人物标为地点
- 结构错误：JSON格式不合法

## 三、质量评估方法

### 3.1 自动化检测

**规则检测**：
- 正则表达式检查格式
- 约束检查（SHACL/SPARQL）
- 推理检测矛盾
- 统计分析异常

**优点**：
- 快速、全面、可重复
- 适合大规模检查
- 成本低

**局限**：
- 只能检测已知规则
- 无法判断语义正确性
- 可能漏检复杂错误

### 3.2 LLM辅助验证

**方法**：
- 让LLM审查标注结果
- 让LLM回答验证性问题
- 让LLM解释推理过程
- 对比LLM与知识库的答案

**优点**：
- 能理解语义
- 发现人工难以察觉的错误
- 可解释性强

**局限**：
- LLM本身可能出错
- 成本较高
- 需要人工最终审核

### 3.3 人工审核

**方法**：
- 抽样检查
- 专家审查重点知识
- 用户反馈错误
- 同行评审

**优点**：
- 最高准确性
- 能发现微妙错误
- 提供改进建议

**局限**：
- 成本高
- 难以全面覆盖
- 主观性

### 3.4 交叉验证

**方法**：
- 多模型对比（GPT-4 vs Claude vs本地模型）
- 多轮标注对比（不同时期的结果）
- 多人标注对比（一致性检查）
- 多源数据对比（不同数据库）

**优点**：
- 提高可信度
- 发现系统性偏差
- 量化不确定性

## 四、质量指标体系

### 4.1 准确率（Precision）

**定义**：标注为正确的知识中，真正正确的比例

**计算**：
```
准确率 = 真正确 / (真正确 + 假正确)
```

**目标**：>95%

### 4.2 召回率（Recall）

**定义**：所有正确知识中，被标注出来的比例

**计算**：
```
召回率 = 真正确 / (真正确 + 假负例)
```

**目标**：>90%

### 4.3 F1-Score

**定义**：准确率和召回率的调和平均

**计算**：
```
F1 = 2 × (准确率 × 召回率) / (准确率 + 召回率)
```

**目标**：>92%

### 4.4 一致性指标

**指标**：
- 逻辑矛盾数量
- 重复冗余数量
- 孤立节点数量

**目标**：趋近于0

### 4.5 完整性指标

**指标**：
- 必填字段填充率
- 关系连接率
- 覆盖度分数（SKILL_08b）

**目标**：>80%

## 六、错误修正策略

### 6.1 错误分类

**A类错误**：严重错误，必须立即修复
- 逻辑矛盾
- 明显事实错误
- 数据损坏

**B类错误**：中等错误，计划修复
- 格式不规范
- 属性缺失
- 关系不完整

**C类错误**：轻微错误，批量处理
- 命名不统一
- 标签缺失
- 样式问题

### 6.2 修正工作流

1. 错误检测
2. 错误分类
3. 修正优先级排序
4. 修正方案设计
5. 批量或单个修正
6. 修正后验证
7. 记录和总结

### 6.3 根因分析

**问题**：为什么会出现这个错误？

**分析维度**：
- 规范不清？
- 工具缺陷？
- 人为疏忽？
- 数据源问题？
- LLM幻觉？

**改进措施**：
- 完善规范
- 优化工具
- 加强培训
- 提升数据质量
- 改进prompt
