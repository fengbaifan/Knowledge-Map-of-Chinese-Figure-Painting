# SKILL 02a: 古籍章节切分与编号 — Purple Numbers与语义分段

> 从《史记》130篇的章节结构化实践中提炼，适用于大规模古籍的篇章拆分、段落编号和小节划分。

---

## 一、核心概念

### 1.1 三层结构

古籍文本需要三层结构化：

```
篇章（Chapter）    130个独立文件，三位数编号
  └─ 小节（Section）  语义段落组，二/三级标题
      └─ 段落（Paragraph）  带Purple Numbers的原子文本单元
```

### 1.2 Purple Numbers（紫色编号）

**定义**：每个段落的唯一、永久、不可变标识符。

```
[0]      篇名段
[1]      第一段
[1.1]    第一段的第一子段
[1.1.1]  三级子段（可选，用于复杂章节）
[2]      第二段
```

**为什么叫"Purple Numbers"**：
- 源自Project Xanadu的概念 — 文档中每个段落都有永久地址
- 一旦分配，永不改变（即使周围内容重组）
- 渲染时显示为可点击的锚点链接

**为什么重要**：
- 跨文档引用：`092_淮阴侯列传#pn-3.1` 永远指向同一段文字
- 实体索引关联：`entity_index.json` 中每个实体引用格式为 `[chapter, paragraph_id]`
- 事件索引关联：事件定位到具体段落
- 搜索定位：精确到段落级别的全文检索

**详细规范参考**：
- [references/SKILL_02a1_Purple_Numbers编号详细规范.md](references/SKILL_02a1_Purple_Numbers编号详细规范.md) — 编号层级、策略选择、父子关系、连续性验证
- [references/SKILL_02a2_回车规则.md](references/SKILL_02a2_回车规则.md) — 段落内换行标准（平行世系、密集列举按句号拆行，叙事保持单行，表格不拆）

## 二、阶段1：篇章拆分

### 2.1 拆分规则

将完整古籍文本按卷/篇拆分为独立文件。

**命名规范**：`{三位编号}_{篇名}.txt`

```
001_五帝本纪.txt
002_夏本纪.txt
...
130_太史公自序.txt
```

**《史记》的体裁分布**：

| 体裁 | 篇数 | 编号范围 | 特征 |
|------|------|---------|------|
| 本纪 | 12 | 001-012 | 帝王编年，时间跨度大 |
| 表 | 10 | 013-022 | 表格数据，结构特殊 |
| 书 | 8 | 023-030 | 制度专题，论述为主 |
| 世家 | 30 | 031-060 | 诸侯/功臣，叙事+编年 |
| 列传 | 70 | 061-130 | 人物传记，叙事为主 |

### 2.2 拆分脚本逻辑

```python
# 识别篇章边界：标题行以体裁关键词结尾
CHAPTER_PATTERNS = ['本纪', '表', '书', '世家', '列传', '太史公自序']

def split_by_chapter(full_text):
    chapters = []
    current = []
    for line in full_text.split('\n'):
        if any(line.strip().endswith(p) for p in CHAPTER_PATTERNS):
            if current:
                chapters.append(current)
            current = [line]
        else:
            current.append(line)
    return chapters
```

**注意事项**：
- 编号补零到3位，确保文件系统排序正确
- 保持原文完整性，不做任何修改
- 表类篇章（013-022）可能需要特殊处理（表格结构）

## 三、阶段1.5：段落修复（误分段合并）

原始文本拆分后，常因源文件换行位置不当而产生误分段（一个句子被截断为两行）。需在段落编号之前修复。

### 误分段特征

```
於是九州攸同，四奥既居，九山     ← 行末无句末标点
旅，九川涤原，九泽既陂...       ← 下一行直接续前句
```

### 合并规则（4条）

| 规则 | 条件 | 操作 | 理由 |
|------|------|------|------|
| 1 | 当前行以句末标点结尾（。！？」』：；…） | 不合并 | 完整句子 |
| 2 | 当前行或下一行为空 | 不合并 | 段落分隔 |
| 3 | 当前行很短（<10字符） | 不合并 | 可能是标题或小节标记 |
| 4 | 其他情况 | 合并 | 行末无句末标点，很可能是误分段 |

### 合并算法

```python
ending_marks = ['。', '！', '？', '」', '』', '：', '；', '…']

def should_merge(current_line, next_line):
    if not current_line.strip() or not next_line.strip():
        return False                          # 规则2
    if len(current_line.strip()) < 10:
        return False                          # 规则3
    if current_line.rstrip()[-1] in ending_marks:
        return False                          # 规则1
    return True                               # 规则4：合并

# 循环合并：当前行可能需要与连续多行合并
merged = current_line
while should_merge(merged, next_line):
    merged += next_line
    # 继续检查下一行
```

### 扩展到其他古籍

句末标点列表需按古籍标点系统调整。未标点的古籍（如原始刻本）不适用此规则，需先完成标点再修复。

## 四、阶段2：段落编号（Purple Numbers）

> **详细规范参考**：本节为快速指南，完整的Purple Numbers编号规范（包括三级编号、编号策略选择、特殊情况处理等）请参考：
>
> [references/SKILL_02a1_Purple_Numbers编号详细规范.md](references/SKILL_02a1_Purple_Numbers编号详细规范.md)

### 4.1 编号格式与规则

#### 4.1.1 基本格式

Purple Number采用**层级点分格式**：

```
[0]        篇名（固定）
[1]        一级段落（顶级）
[1.1]      二级段落（子段落）
[1.1.1]    三级段落（子子段落，可选）
[2]        一级段落
[2.1]      二级段落
```

**格式要求**：
- 方括号 `[]` 包裹
- 纯数字，用 `.` 分隔层级
- 行首位置，紧跟一个空格
- 不允许字母或中文：❌ `[1.a]`、❌ `[一]`
- 支持最多三级编号（复杂章节可使用三级结构）

#### 4.1.2 父子关系规则（重要）

**核心规则**：必须有父段落才能有子段落。

❌ **错误示例**（缺少父段落）：
```markdown
## 某节标题
[67.1] 子段落内容...
[67.2] 子段落内容...
```

✅ **正确示例**（有父段落）：
```markdown
## 某节标题
[67]               ← 父段落（可以为空）
[67.1] 子段落内容...
[67.2] 子段落内容...
```

**空父段落是允许的**：
- 当一个小节下全是子段落时，可以只有编号 `[67]`，无内容
- 父段落作为子段落的"容器"存在

**检查与修复工具**：

```bash
# 检查缺失的父段落编号
python scripts/check_missing_parent_pn.py

# 自动修复（插入缺失的父段落）
python scripts/fix_missing_parent_pn.py
```

### 4.2 编号的连续性

#### 4.2.1 同层级必须连续

**规则**：同层级编号不允许跳跃。

❌ **错误**：
```markdown
[1] 第一段
[3] 第三段    ← 缺少[2]
[3.1] 子段
[3.3] 子段    ← 缺少[3.2]
```

✅ **正确**：
```markdown
[1] 第一段
[2] 第二段
[3] 第三段
[3.1] 子段
[3.2] 子段
```

#### 4.2.2 检查工具

`lint_markdown.py` 会检查编号连续性，报告跳跃的编号。

### 4.3 何时使用子编号

#### 4.3.1 长段落拆分

**原则**：一段超过500字，按语义边界拆分为子段落。

```markdown
[5]                        ← 空的父段落
[5.1] 第一个语义单元...
[5.2] 第二个语义单元...
[5.3] 第三个语义单元...
```

#### 4.3.2 对话段落

**原则**：包含多人对话时，每句对话独立编号。

```markdown
[12]
- [12.1] @尧@曰："谁可顺此事？"
- [12.2] @放齐@曰："嗣子@丹硃@开明。"
- [12.3] @尧@曰："吁！顽凶，不用。"
```

详细规则见 §8.2 对话段落。

#### 4.3.3 列表项处理

**短列表**（方位、物品名等）：不需要子编号

```markdown
[2]
- 东至于=海=，登=丸山=，及=岱宗=。
- 西至于=空桐=，登=鸡头=。
- 南至于=江=，登=熊=、=湘=。
```

**长列表**（完整句子、对话）：需要子编号

```markdown
[12]
- [12.1] @尧@曰："谁可顺此事？"
- [12.2] @放齐@曰："嗣子@丹硃@开明。"
```

#### 4.3.4 引用块

**原则**：引用块（`> "..."`）不独立编号，从属于上一段。

```markdown
[1] 太史公曰：余登箕山，其上盖有许由冢云。

> "余以所闻由光义至高，其文辞不少概见，何哉？"

[2] 孔子序列古之仁圣贤人...
```

### 4.4 编号的不可变性

**核心原则**：Purple Numbers一旦分配，永不更改。

**为什么**：
- 外部引用依赖编号：实体索引中每个实体的 `paragraph_id` 字段
- 事件索引依赖编号：事件的段落位置
- 跨文档引用：`092_淮阴侯列传#pn-3.1` 永远指向同一段

**插入新段落时**：使用子编号，不改变已有编号

```markdown
[3] 原有段落
[3.1] 后续插入的段落（新增）
[4] 原有段落（编号不变）
```

### 4.5 特殊编号格式

#### 4.5.1 篇名编号

**固定格式**：每篇开头的篇名固定使用 `[0]`

```markdown
# [0] 五帝本纪

[1] 黄帝者，少典之子...
```

#### 4.5.2 表格行编号（十表专用）

**格式**：表格行使用 `[rN]` 格式（`r` = row）

```markdown
| 国名 | 侯功 | 元光 | 元朔 |
| --- | --- | --- | --- |
| [r1] 翕。 | 匈奴相降，侯。 | 三。四年七月壬午... |
| [r2] 持装。 | 匈奴都尉降，侯。 | ... |
```

详见 §8.3 表类篇章。

### 4.6 编号放置位置

**标准格式**：`[编号]` + 空格 + 内容

```markdown
[1] 太史公曰：余登箕山，其上盖有许由冢云。

[1.1] 孔子序列古之仁圣贤人，如吴太伯、伯夷之伦详矣。
```

**列表项格式**：`- ` + `[编号]` + 空格 + 内容

```markdown
- [12.1] @尧@曰："谁可顺此事？"
- [12.2] @放齐@曰："嗣子@丹硃@开明。"
```

## 五、阶段3：语义小节划分

### 4.1 小节层级

```markdown
# [0] 篇名                    ← 一级标题（每篇唯一）

## 大节标题                    ← 二级标题（主要叙事段落）

### 小节标题                   ← 三级标题（细分主题）

[1.1] 正文段落...
```

### 4.2 分节标题规则

#### 4.2.1 标题不应包含段落编号

**重要规则**：二级和三级标题（##、###）不应包含段落编号。

**错误示例**：

```markdown
## [6] 论法家        ← 错误！标题不应包含编号
### [2.1] 历法起源   ← 错误！标题不应包含编号
```

**正确示例**：

```markdown
## 论法家           ← 正确！标题只包含文字
### 历法起源        ← 正确！标题只包含文字
```

**例外**：一级标题 `# [0] 篇名` 可以包含编号（用于篇章标识）。

**检查与修复工具**：

```bash
# 检查标题中的编号
python scripts/lint_heading_numbers.py

# 自动修复（去除标题中的编号）
python scripts/fix_heading_numbers.py
```

#### 4.2.2 标题位置规则

**重要规则**：分节标题（##、###）必须出现在顶级段落之前，不应出现在非顶级段落（子段落）之前。

**错误示例**：

```markdown
[4] 黄帝二十五子，其得姓者十四人。
### 后裔与世系    ← 错误！标题在子段落之前
[4.1] 黄帝居轩辕之丘...
```

**正确示例**：

```markdown
### 后裔与世系    ← 正确！标题在顶级段落之前
[4] 黄帝二十五子，其得姓者十四人。
[4.1] 黄帝居轩辕之丘...
```

**修复方法**：

1. 如果顶级段落（如[4]）已存在，将标题移到顶级段落之前
2. 如果顶级段落不存在（如缺少[2]，直接从[2.1]开始），在标题后、子段落前插入空的顶级段落：

```markdown
### 秦国疆域与大臣
[2]           ← 插入空的顶级段落
[2.1] 当是之时...
```

**检查与修复工具**：

```bash
# 检查分节位置错误
python scripts/lint_section_position.py

# 自动修复（插入缺失的顶级段落）
python scripts/fix_section_position.py
```

### 4.4 两种小节格式

**格式A：标题带编号**

```markdown
## [1] 早年困顿
[1.1] @淮阴侯@韩信@者，=淮阴=人也...
[1.2] @信@钓於城下...

## [2] 投奔项羽与汉王
[2.1] 及@项梁@渡=淮=...
```

锚点直接取自标题中的编号：`#pn-1`、`#pn-2`。

**格式B：标题不带编号**

```markdown
## 项羽早年
### 项氏家世
[1.1] @项籍@者，=下相=人也...

## 起兵反秦
### 斩会稽守
[7.1] &秦&%二世元年七月%...
```

锚点取自该节第一个段落的编号：`#pn-1.1`、`#pn-7.1`。

### 4.5 按体裁定制小节策略

| 体裁 | 划分原则 | 典型小节 |
|------|---------|---------|
| 本纪 | 按帝王/时期 | "黄帝"、"帝颛顼"、"帝喾" |
| 世家 | 按诸侯世代 | "太伯奔吴"、"寿梦立"、"阖闾争位" |
| 列传（单传）| 按人生阶段 | "早年"、"拜将"、"封王"、"陨落" |
| 列传（合传）| 按人物分段 | "管仲"、"晏婴" |
| 列传（类传）| 按人物分段 | "总论"、"循吏某某"、"循吏某某" |
| 书 | 按制度主题 | "礼之起源"、"三代之礼"、"秦汉之变" |
| 表 | 按表格结构 | 保留原表，可能需特殊处理 |

### 4.6 小节标题规范

**好的标题**：
- "鸿门宴" — 具体事件
- "萧何月下追韩信" — 经典典故
- "背水之战" — 著名战役

**差的标题**：
- "第一部分" — 无信息量
- "段落1-10" — 纯位置描述
- "内容" — 过于笼统

**标题中清理实体标记**：小节标题在 `sections_data.json` 中存储时，去除所有实体标记符号：

```python
def clean_title(title):
    """去除标题中的实体标记"""
    for ch in ['@', '=', '$', '%', '&', '^', '~', '*', '!', '〖+', '〗', '〖?', '#']:
        title = title.replace(ch, '')
    return title.strip()
```

### 4.7 自动小节生成

对于没有明确分段的章节（约41篇），使用启发式算法生成建议：

```python
def auto_generate_sections(paragraphs, chunk_size=10):
    """每10段为一组，分析前3段内容生成标题"""
    sections = []
    for i in range(0, len(paragraphs), chunk_size):
        chunk = paragraphs[i:i+chunk_size]
        sample = ' '.join(chunk[:3])
        title = infer_title(sample)
        sections.append({"anchor": chunk[0].pn, "title": title})
    return sections

def infer_title(text):
    """基于关键词的启发式标题推断"""
    if '曰' in text or '云' in text: return '评论与论述'
    if '年' in text or '月' in text: return '编年记事'
    if '战' in text or '伐' in text: return '战争与征伐'
    if '封' in text or '侯' in text: return '封赏与爵位'
    if '太史公' in text: return '太史公评论'
    return '叙事'  # 默认
```

**重要**：自动生成的标题仅为建议，必须人工审核后使用。


**检查项**：
1. 编号格式检查：`[N]` 或 `[N.N]` 格式，不允许字母或中文
2. 编号连续性检查：同层级不允许跳跃（`[1]→[2]→[3]`，不允许`[1]→[3]`）
3. 父子关系检查：子编号必须有对应的父编号（有`[67.1]`必须有`[67]`）
4. 标题编号检查：二级和三级标题不应包含编号
5. 重复编号检查：同一章内不允许重复编号
6. 篇名编号检查：一级标题中必须有`[0]`

#### 7.1.2 专项检查工具

| 脚本 | 功能 | 使用场景 |
|------|------|---------|
| `check_missing_parent_pn.py` | 检查缺失的父段落编号 | 只检查父子关系 |
| `lint_heading_numbers.py` | 检查标题中的编号 | 只检查标题问题 |
| `lint_section_position.py` | 检查分节位置错误 | 检查标题与段落位置关系 |

#### 7.1.3 自动修复工具

| 脚本 | 功能 | 使用方法 |
|------|------|---------|
| `fix_missing_parent_pn.py` | 插入缺失的父段落编号 | `python scripts/fix_missing_parent_pn.py` |
| `fix_heading_numbers.py` | 去除标题中的编号 | `python scripts/fix_heading_numbers.py` |
| `fix_section_position.py` | 修复分节位置错误 | `python scripts/fix_section_position.py` |

### 7.2 小节处理工具

## 八、质量检查

### 8.1 Purple Numbers完整性检查

**推荐工具**：`lint_purple_numbers.py`（整合所有编号检查）

```bash
# 检查所有章节
python scripts/lint_purple_numbers.py

# 检查指定章节
python scripts/lint_purple_numbers.py 092
```

**六大检查项**：

#### 9.1.1 编号格式检查

- ✓ 正确格式：`[1]`、`[1.2]`、`[1.2.3]`
- ✗ 错误格式：`[1.a]`、`[一]`、`[1-2]`

#### 9.1.2 编号连续性检查

- ✓ 正确：`[1] → [2] → [3]`
- ✗ 错误：`[1] → [3]`（缺少[2]）
- ✓ 正确：`[3.1] → [3.2] → [3.3]`
- ✗ 错误：`[3.1] → [3.3]`（缺少[3.2]）

#### 9.1.3 父子关系检查

- ✓ 正确：有`[67]`，才能有`[67.1]`、`[67.2]`
- ✗ 错误：没有`[67]`，直接从`[67.1]`开始

**修复**：`python scripts/fix_missing_parent_pn.py`

#### 9.1.4 标题编号检查

- ✓ 正确：`## 标题文字`
- ✗ 错误：`## [6] 标题文字`（二级标题不应包含编号）
- ✓ 例外：`# [0] 篇名`（一级标题可以有`[0]`）

**修复**：`python scripts/fix_heading_numbers.py`

#### 9.1.5 重复编号检查

- ✗ 错误：同一章内出现两个`[3.1]`
- **修复**：手动检查并重新编号

#### 9.1.6 篇名编号检查

- ✓ 正确：`# [0] 篇名`
- ✗ 错误：`# [1] 篇名`或`# 篇名`（无编号）

### 8.2 常见问题及修复方式

| 问题类型 | 检测方法 | 修复方式 |
|---------|---------|---------|
| 编号格式错误 | `lint_purple_numbers.py` | 手动修改为标准格式 |
| 编号跳跃 | `lint_purple_numbers.py` | 检查是否遗漏段落或编号错误 |
| 缺少父编号 | `check_missing_parent_pn.py` | `fix_missing_parent_pn.py` 自动修复 |
| 标题包含编号 | `lint_heading_numbers.py` | `fix_heading_numbers.py` 自动修复 |
| 重复编号 | `lint_purple_numbers.py` | 手动检查并重新编号 |
| 篇名编号错误 | `lint_purple_numbers.py` | 手动确认并修改 |
| 标题层级跳跃 | `lint_markdown.py` | 补充缺失的标题层级 |

## 九、特殊情况处理

### 9.1 "赞"段落（太史公评论）

每篇末尾通常有"太史公曰"或"赞"段落，格式特殊：

```markdown
## 赞
[N] 太史公曰：...评论文字...
多行评论合并为一个段落编号
不需要每行单独编号
```

**修复工具**：`fmt_fix_zan.py` — 将多编号的赞合并为单编号，各句换行但不分段。

### 9.2 对话段落

包含多人对话的长段落需要拆分，拆分后每行转为列表项（`- ` 开头）：

```markdown
# 拆分前：
[5] 齐王曰："寡人闻之。"晏子对曰："臣请言之。"

# 拆分后：
- [5.1] 齐王曰："寡人闻之。"
- [5.2] 晏子对曰："臣请言之。"
```

**修复工具**：`fmt_split_dialogues.py` — 在"X曰:"边界处自动拆分。

详细规则与示例见 [SKILL_02a_章节切分与编号.md §五](SKILL_02a_章节切分与编号.md)。

#### 9.2.1 嵌套引号的拆分规则

将包含嵌套引号的长段落拆分为列表项时，**必须保持原文的引号层级**，不得将内层单引号 `'...'` 改为外层双引号 `"..."`。

```markdown
# 原文（一整段）：
作汤诰："维三月，王自至於东郊。告诸侯群后：'毋不有功於民，勤力乃事。予乃大罚殛女，毋予怨。'曰：'古禹、皋陶久劳于外，...先王言不可不勉。'曰：'不道，毋之在国，女毋我怨。'"以令诸侯。

# ✗ 错误拆分（内层引号被改为双引号）：
- [8.1] "维三月，...毋予怨。'"
- [8.2] 曰："古禹、皋陶...不可不勉。"    ← 错：'→"
- [8.3] 曰："不道，...女毋我怨。"          ← 错：'→"

# ✓ 正确拆分（保持内层单引号）：
- [8.1] "维三月，...毋予怨。'
- [8.2] 曰：'古禹、皋陶...不可不勉。'
- [8.3] 曰：'不道，...女毋我怨。'"
```

**原则**：拆分只改变物理换行，不改变原文的任何字符（包括引号类型）。外层 `"` 的开闭仍然在首段和末段，内层 `'` 保持不变。

### 9.3 表类篇章（十表 013-022）

年表（013-022）以表格形式呈现，每行数据对应一个历史条目（侯国、诸侯、君主、将相等），**必须为每行分配独立 Purple Number**，以支持精确引用和事件链接。

#### 9.3.1 行号格式

表格行使用 `[rN]` 格式（`r` 前缀区别于段落编号 `[1.1]`），章内全局递增，跨多张表格连续计数：

```markdown
| 国名 | 侯功 | 元光 | 元朔 | … |
| --- | --- | --- | --- | … |
| [r1] 翕。 | 匈奴相降，侯。 | 三。四年七月壬午，侯赵信元年。 | … |
| [r2] 持装。 | 匈奴都尉降，侯。 | … |
| [r5] 长平。 | 以元朔二年再以车骑将军击匈奴… | … |
```

渲染器读取 `[rN]` 并生成锚点 `id="pn-r5"`，HTML URL 为 `020_建元以来侯者年表.html#pn-r5`。

#### 9.3.2 生成脚本

`scripts/add_table_row_pn.py` 可一键为十表 MD 文件添加 `[rN]` 标记：

```bash
python scripts/add_table_row_pn.py          # 处理全部 013-022 章
python scripts/add_table_row_pn.py 020       # 只处理 020 章
```

已有 `[rN]` 的行会自动跳过，支持幂等运行。

#### 9.3.3 在事件索引中引用

十表来源的事件，`段落位置` 字段应使用 `r{N}` 格式，而非 `表（X条目）`：

```markdown
- **段落位置**: r5          ← 对应 #pn-r5（长平侯条目）
- **段落位置**: r8          ← 对应 #pn-r8（平津侯/公孙弘条目）
```

多行事件可记为范围 `r12-r15`。

#### 9.3.4 CSS 渲染

行号列（`<td class="row-pn-col">`）样式在 `docs/css/shiji-styles.css` 中定义，与段落 `.para-num` 使用相同颜色（淡紫色 `#C8A8FF`）。表格其余样式（全视口宽度、sticky 表头、间隔行色）由 `.shiji-table-wrapper` / `.shiji-table` / `.even` / `.odd` 控制，也在同一 CSS 文件中。

### 9.4 超长小节

如果一个小节超过20个段落，应进一步拆分：

```python
MAX_SECTIONS_PER_CHAPTER = 15  # 小节上限
MIN_PARAGRAPHS_PER_SECTION = 3  # 太短的小节应合并
MAX_PARAGRAPHS_PER_SECTION = 20 # 太长的小节应拆分
```

## 十一、经验教训

1. **编号是基础设施** — Purple Numbers是整个知识系统的定位基础（实体索引、事件索引、跨文档引用都依赖它），设计必须在最早期确定且不可变。

2. **语义分段优于机械分段** — "每500字切一段"远不如"按帝王更迭切一段"有价值。小节标题是读者的导航地图。

3. **体裁决定结构** — 本纪按时间、列传按人生、世家按世代，不同体裁的最佳分段策略完全不同。

4. **自动生成只是起点** — 启发式规则生成的小节标题准确率约60%，必须人工审核。但它大幅降低了从零开始的工作量。

5. **小节数量要克制** — 每篇5-15个小节是合理范围。太多（>20）等于没有分段，太少（<3）等于只有标题。

6. **两种标题格式都要支持** — 有些章节天然适合"## [N] 标题"格式（列传），有些适合无编号格式（本纪），工具链必须兼容两者。

7. **锚点格式统一** — `#pn-{number}` 作为唯一锚点格式贯穿整个系统（源文件、HTML、索引、API），避免多套ID体系。

8. **先切分后标注** — 先完成篇章拆分和段落编号，再做实体标注。如果顺序反过来，每次修改编号都要重新关联所有实体引用。
