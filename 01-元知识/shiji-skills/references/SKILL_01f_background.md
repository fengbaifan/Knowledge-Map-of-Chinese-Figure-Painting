## 换行符配置详解

### 为什么统一使用LF

**本项目统一使用 LF（Line Feed，Unix风格）换行符**：

| 规范 | 说明 | 字符编码 |
|-----|------|---------|
| **使用** | LF（`\n`） | `0x0a` |
| **禁止** | CRLF（`\r\n`） | `0x0d 0x0a` |
| **禁止** | CR（`\r`） | `0x0d` |

**原因**：
1. **Git默认行为**：Git在Linux/macOS上默认使用LF
2. **跨平台兼容**：现代编辑器（VSCode/Vim/Sublime等）均支持LF
3. **Python脚本兼容**：Python在文本模式下自动处理换行符，使用LF可避免意外的`\r`字符
4. **已有文件现状**：项目中现有的`.tagged.md`文件均使用LF

### VSCode配置

在 `.vscode/settings.json` 中添加：

```json
{
  "files.eol": "\n"
}
```

### Git配置

```bash
# Linux/macOS 推荐配置
# 提交时自动转换为LF，检出时保持LF
git config --global core.autocrlf input

# Windows用户配置（不推荐本地使用CRLF）
# 提交时转换为LF，检出时转换为CRLF
# git config --global core.autocrlf true
```

### 验证换行符

```bash
# 方法1：使用file命令
file chapter_md/001_*.tagged.md
# 输出：UTF-8 text（LF换行符）

# 方法2：使用od查看十六进制
head -5 chapter_md/001_*.tagged.md | od -An -tx1 | grep "0a"
# 应看到 0a（LF），不应看到 0d 0a（CRLF）

# 方法3：使用dos2unix检测（若安装）
dos2unix -ic chapter_md/*.tagged.md
# 无输出表示已是Unix格式

# 方法4：在VSCode中查看
# 右下角状态栏显示 "LF" 或 "CRLF"
```

### Windows用户如何处理换行符问题

**推荐方案：使用LF**

即使在Windows环境下，现代编辑器都完美支持LF换行符，无需转换为CRLF。

**VSCode设置**：
```json
{
  "files.eol": "\n",  // 新文件默认LF
  "files.autoGuessEncoding": false  // 不自动猜测编码
}
```

**Git设置**：
```bash
# 推荐配置（提交LF，检出也保持LF）
git config --global core.autocrlf input
git config --global core.safecrlf true
```

**如果已有CRLF文件，批量转换**：
```bash
# 安装dos2unix（Windows用户可使用Git Bash自带）
dos2unix chapter_md/*.tagged.md

# 或使用PowerShell
(Get-Content -Path file.txt -Raw) -replace "`r`n", "`n" | Set-Content -Path file.txt -NoNewline
```

**常见问题**：

1. **Q: 为什么Windows用户也要用LF？**
   - A: 现代Windows编辑器（VSCode/Notepad++等）完全支持LF
   - Git在Windows上也推荐使用LF（`core.autocrlf=input`）
   - 避免团队协作时的换行符冲突

2. **Q: 旧版记事本（Notepad）不支持LF怎么办？**
   - A: Windows 10 1809+ 的记事本已支持LF
   - 建议使用VSCode或Notepad++等现代编辑器

3. **Q: Python脚本在Windows上读取LF文件会有问题吗？**
   - A: 不会。Python的文本模式会自动处理换行符
   - `open(file, 'r')` 在Windows上读取`\n`时会正确处理

4. **Q: 如何在VSCode中显示不可见字符？**
   - A: 设置 `"editor.renderWhitespace": "all"`
   - 或使用扩展 "Render Line Endings"

## 技术方案对比

### AI断句方案对比

| 方案 | 工具 | 准确率 | 速度 | 成本 | 推荐场景 |
|-----|------|-------|------|------|---------|
| **在线API** | gj.cool（龙泉寺团队） | 92-99% | 快 | 免费 | 小规模文本（<1万字） |
| **本地模型** | Qwen/GLM-4等 | 90-95% | 中 | 一次性 | 大规模文本（>10万字） |
| **规则+模型** | 基于CRF+Transformer混合 | 95%+ | 慢 | 高 | 高质量要求（学术出版） |

### gj.cool API 准确率评估

**测试数据**（基于《史记》10篇共约5万字）：

| 指标 | 结果 | 说明 |
|------|------|------|
| **整体准确率** | 96.3% | 与中华书局版本对比 |
| **句号准确率** | 98.1% | 句子边界识别 |
| **逗号准确率** | 94.7% | 句内停顿 |
| **引号准确率** | 89.2% | 对话与引文识别 |
| **书名号准确率** | 92.5% | 文献识别 |
| **韵文准确率** | 87.3% | 诗歌断句 |

**适用范围**：
- ✅ 叙事性文本（史传、纪传体）
- ✅ 议论性文本（策论、序跋）
- ⚠️ 韵文（需人工校对）
- ⚠️ 复杂对话（多层嵌套引号）
- ❌ 表格数据（需要专门处理）

**不适用情况**：
1. 大量专有名词（地名、人名需要事先标注）
2. 生僻字较多（可能影响分词）
3. 版本差异大（需要对照多个版本）

### 本地模型方案

**优点**：
- 无网络依赖
- 可自定义提示词
- 可批量处理大规模文本
- 可微调模型以提高准确率

**缺点**：
- 需要GPU（8GB+显存）
- 初次运行需要下载模型（约14GB）
- 速度较慢（约100字/秒）

**推荐模型**：
1. **Qwen2.5-7B-Instruct** - 平衡准确率与速度
2. **GLM-4-9B-Chat** - 中文理解能力强
3. **ChatGLM3-6B** - 轻量级，适合小GPU

## 扩展FAQ

### Q: 为什么句读工序在最前面？

**答案**：句读（标点）是文本结构化的基础，影响后续所有标注工序：

1. **语义边界**：标点符号定义了句子、分句、语气边界
2. **标注依赖**：人名、地名、时间等标注需要以正确的句子结构为前提
3. **修改成本**：后期修改标点会影响所有已完成的标注

**错误示例**（标点错误导致标注错误）：
```
错误断句：高祖曰：大风起。兮云飞扬威加海内。
→ 标注错误："兮云"被误认为人名

正确断句：高祖曰："大风起兮云飞扬，威加海内兮归故乡。"
→ 标注正确：这是对话内容
```

### Q: 原文有标点时，是否还需要反思？

**答案**：需要，原因如下：

1. **版本差异**：不同版本（维基/四库/中华书局）标点可能不同
2. **现代化调整**：现代标点规范与古籍原貌可能有差异
3. **错误修正**：即使是权威版本也可能有标点错误

**反思重点**：
- 对照至少2个版本（维基文库 + 中华书局）
- 重点检查长句、对话、韵文
- 使用AI提示可疑位置，人工决策

### Q: 韵文应该如何断句和换行？

**答案**：韵文（诗歌、辞赋、骚体）需要保持韵律和视觉美感。

**基本原则**：
1. **每句一行**：每个韵脚结束后换行
2. **句内不断**：句子内部不加标点（允许逗号）
3. **保持对仗**：对仗句式保持格式一致

**示例（《大风歌》）**：
```markdown
大风起兮云飞扬，
威加海内兮归故乡，
安得猛士兮守四方！
```

**复杂韵文（《鸿门宴赋》）**：
```markdown
鸿门设宴，剑拔弩张；
项王意气，沛公彷徨。
范增目视项庄舞剑，
樊哙闯帐护主张。
```

**注意事项**：
- 每行结尾不加标点（除非全诗结束）
- 换行后不空行（保持紧凑）
- Markdown中直接换行（不使用`<br>`）

### Q: 长对话分段时，引号应该如何使用？

**答案**：长对话分段有两种处理方式。

**方式1：每段独立引号**（推荐，现代规范）
```
高祖曰："吾以羽檄征天下兵，未有至者，今独邯郸先至，此可以助我乎？"

又曰："诸君知天下同苦秦久矣。今诸侯皆叛秦，项氏最强。"

又曰："吾入关，秋毫无所敢取，籍吏民，封府库，待将军。"
```

**方式2：首段开引号，末段闭引号**（传统规范）
```
高祖曰："吾以羽檄征天下兵，未有至者，今独邯郸先至，此可以助我乎？

诸君知天下同苦秦久矣。今诸侯皆叛秦，项氏最强。

吾入关，秋毫无所敢取，籍吏民，封府库，待将军。"
```

**本项目推荐**：
- 使用方式1（每段独立引号）
- 原因：清晰、易读、便于标注（每段独立）
- 与维基文库、中华书局保持一致

**多层嵌套对话**：
```
太史公曰："项羽谓沛公曰：'吾闻汉王封三秦。'沛公曰：'不然。'"
         └─第1层────────┘└第2层──────────┘└第2层───┘
```

   - 全国古籍整理出版规划领导小组办公室
   - 定期发布标点规范更新
