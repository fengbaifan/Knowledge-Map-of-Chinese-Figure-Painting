# SKILL 10b: 每日工作日志维护

> **理念**：日志记录项目记忆，微信通知传递进展，改动意义提炼价值。

---

## 一、工作日划分规则

**⚠️ 重要：X日日志计时规则**

- **`YYYY-MM-DD.md` 日志包含的时间范围**：`YYYY-MM-DD 07:00` ~ `YYYY-MM-DD+1 07:00`
- **例如**：`2026-03-30.md` = `2026-03-30 07:00` ~ `2026-03-31 07:00` 的所有提交
- **以早上7:00为分界线**（非传统的0:00）

**原因**：凌晨0-7点通常是前一天延续，以7点为界符合作息，便于统计，避免深夜工作被切分。

**时间归属示例**：

| 提交时间 | 计入日志 | 说明 |
|---------|---------|------|
| 2026-03-23 08:00 | 2026-03-23.md | 白天工作算当日 |
| 2026-03-23 23:30 | 2026-03-23.md | 深夜工作算当日 |
| 2026-03-24 02:00 | 2026-03-23.md | 凌晨算前一日 |
| 2026-03-24 06:59 | 2026-03-23.md | 7点前算前一日 |
| 2026-03-24 07:00 | 2026-03-24.md | 7点起算次日 |

## 三、微信群通知生成

**流程**：读取日志 → 检查提交次数 → 提取核心工作 → 生成通知 → 更新日志文件
- 有提交 → 常规通知（3.2）
- 无提交 → 太史公曰（3.3）

### 3.2 有提交日的通知格式

```
【史记知识库 YYYY-MM-DD】

· 核心工作1（包含关键数字）
· 核心工作2（包含关键数字）
· 核心工作3
...
· 提交N次代码

为什么做这些事？
（背景和动机说明，1-3句话）

要解决什么问题？
（痛点和目标说明，1-3句话）

会影响什么最终交付物？
（成果和价值说明，1-3句话）
```

**要求**：
- **格式**：全角中点（·），纯文本，无markdown加粗/章节符号，6-9项工作（最多10项）
- **语言**：平实易懂，避免技术黑话，简化版本号，转化为成果描述
- **内容**：尽量包含数字，最后一项固定为"提交N次代码"
- **优先级**：带数字 > 完成性工作 > 新增功能 > 核心模块
- **改动意义**：三问各1-3句话，总计150-200字，全局视角
- **⚠️ 换行格式**：每个问题标题后必须换行，再写回答内容（问题标题和回答之间有空行）

**技术术语简化示例**：

| 原文 | 简化后 |
|------|--------|
| v2.8格式统一：18类实体全部迁移为〖TYPE X〗格式 | 统一18类实体标注格式 |
| 内联消歧语法 v2.7：扩展到所有13种实体类型 + lint修复 | 消歧语法扩展到13类实体 |
| 重建HTML/实体索引，更新反思文档和SKILL规则 | 更新实体索引和反思文档 |
| 人名实体跨章反思：615处修正（邦国/氏族/动词/边界损坏） | 跨章反思修正615处人名标注 |

### 3.3 无提交日的太史公曰

```
【史记知识库 YYYY-MM-DD】

太史公曰：是日无所书。

<32字赞文，8个四字句>
```

**赞文规范**（8句×4字=32字，仅计汉字）：
- **句式**：必须全部四字句，禁用三/五/六字句
- **文风**：参考 `labs/sima-qian-style/SKILL.md`，对仗/排比，简短有力，尽量押韵
- **主题**：项目建设/持续积累/休整思考/工程浩大
- **禁止**：现代白话（"的""了""很"），重复正文
- **变化**：每天不同主题和句式

**示例**：
```
休整：大业未成，日复一日；张弛有度，蓄力待时。善于休整，方能长久；厚积薄发，终有所成。
思考：代码暂止，思虑不息；磨刀不误，砍柴之工。三思后行，智者所为；谋定后动，方克有济。
持续：百卷宏业，非一日成；偶有停歇，理之常然。持之以恒，乃成大器；锲而不舍，金石可镂。
```

## 四、改动意义分析

**改动意义已整合到微信通知代码块中**（见3.2节格式），回答三问（150-200字）：
1. **为什么做这些事？** - 动机和背景
2. **要解决什么问题？** - 痛点和目标
3. **会影响什么最终交付物？** - 成果和价值

**要求**：平实直白，避免术语，全局视角，简洁有力，纯文本无markdown语法

**场景提示**：
- **无提交日**：简明扼要
- **修复错误**：说明发现问题、建立机制、提升质量
- **开发工具**：说明手工局限、自动化方案、长期价值
- **文档重构**：说明演变背景、规范建立、导航提升

## 五、完整实施流程

```
生成日志 → 读取分析 → 检查提交 → 生成通知/意义 → 更新文件 → 确认输出
           (有提交→常规通知, 无提交→太史公曰)
```

**详细步骤**：

1. **生成日志**：`python logs/daily/generate_log.py YYYY-MM-DD`

2. **统计关键数据**（数字说话）：
   ```bash
   # 统计涉及的章节数
   git log --since="YYYY-MM-DD 07:00" --until="YYYY-MM-DD+1 07:00" \
     --name-only --format="" | grep "chapter_md.*tagged.md" | sort -u | wc -l

   # 统计涉及的总文件数
   git log --since="YYYY-MM-DD 07:00" --until="YYYY-MM-DD+1 07:00" \
     --name-only --format="" | sort -u | wc -l

   # 查找新增的SKILL/机制
   git log --since="YYYY-MM-DD 07:00" --until="YYYY-MM-DD+1 07:00" \
     --diff-filter=A --name-only | grep -E "SKILL|README|SPEC"
   ```

3. **理解工作重点**（不要依赖自动分类）：
   - ❌ 不要直接使用 generate_log.py 的"核心功能变化"
   - ✅ 用实际数据验证（如44章 not 8章）
   - ✅ 识别新增能力（SKILL体系、机制建立）
   - ✅ 询问用户确认核心工作

4. **按战略价值排序**（优先级）：
   - 新能力建设（SKILL、机制）> 流程改进（Issue/TODO）> 质量提升（校对）> 文档完善

5. **生成通知**：
   - 有提交：选5-9条，用实际数字，最后"提交N次代码"
   - 无提交：太史公曰+32字赞文

6. **撰写意义**：回答三问（为什么/解决什么/影响什么），150-200字

7. **更新文件**：在日志最前面插入通知和意义

**插入格式**（在日志标题后）：
```markdown
## 微信群通知

```
【史记知识库 YYYY-MM-DD】

· 核心工作1
· 核心工作2
...
· 提交N次代码

为什么做这些事？

（背景和动机）

要解决什么问题？

（痛点和目标）

会影响什么最终交付物？

（成果和价值）
```

---

## 核心功能变化（原有内容）
```

**注意**：
- 所有内容（工作列表+改动意义）都在代码块内，纯文本格式，便于复制到微信
- ⚠️ **换行格式**：每个问题标题后必须空一行，再写回答内容

## 六、示例（2026-03-29）

**输入**（日志摘要）：
- 新增SKILL_10项目管理4个技能（2500+行）
- 整合6个REF类GitHub Issues
- 修复30+处标注错误（8个章节）
- 001-004章多版本互校同步
- 29个日志补充改动意义
- 提交23次，174个文件

**输出**（微信通知）：
```
【史记知识库 2026-03-29】

· 建立SKILL_10项目管理体系（TODO/Issue/日志/Git/CHANGELOG）
· 整合6个REF类GitHub Issues到参考资源库
· 修复P0级标注错误30+处（8个章节）
· 完成001-004章多版本互校并同步派生文件
· 为29个工作日志补充"改动意义"章节
· 禁止嵌套标注规范（SKILL和Script双层防护）
· 新增SKILL_01a标注完整性维护技能
· 校对基准统一到corpus/archive/chapter目录
· 提交23次代码

为什么做这些事？
项目管理机制分散（TODO、Issue、日志、Git、CHANGELOG各自为政），缺乏系统化SKILL指导。GitHub上6个REF类Issue未整合到参考资源库。多个章节存在P0级标注错误（嵌套标注、地名误标、动词遗漏等）。29个工作日志缺少"改动意义"分析。001-004章互校完成但未同步派生文件。

要解决什么问题？
建立SKILL_10项目管理五件套（TODO/Issue管理、日志维护、Git规范、CHANGELOG编写），整合REF类Issues到resources/references，修复P0级标注错误保证数据质量，为所有日志补充价值分析，同步多版本互校到全流程文件，统一校对基准目录。

会影响什么最终交付物？
SKILL_10体系成为可复用的项目管理方法论（2500+行文档），REF类Issue整合完善参考资源库导航，P0错误修复提升知识图谱准确性，29个日志的改动意义为CHANGELOG提供价值视角，多版本互校同步保证全流程一致性，基准目录统一为后续130章互校奠定基础。
```

## 九、常见错误与避坑指南

### 9.1 不要依赖自动分类结果

**错误案例**（2026-03-29）：
- ❌ 看到"修复8个章节"就写"完成8章校对"
- ✅ 实际统计发现是**44章**（174个文件）

**正确做法**：
```bash
# 用git直接统计，不信任脚本分类
git log --since="..." --name-only | grep "chapter_md.*tagged.md" | sort -u | wc -l
```

### 9.2 理解工作的战略优先级

**错误排序**：
1. 为29个日志添加改动意义（文档完善）
2. 修复8个章节标注错误（质量提升）
3. 建立项目管理体系（能力建设）

**正确排序**：
1. 建立SKILL_10项目管理体系（**能力建设**）← 最重要
2. 整合Issue管理机制（**流程改进**）
3. 完成44章校对（**质量提升**）
4. 补充日志改动意义（**文档完善**）

**原则**：新能力 > 流程改进 > 质量提升 > 文档完善

### 9.3 用数字验证结论

**检查清单**：
- [ ] 章节数是否用git统计验证？
- [ ] 是否识别出新增的SKILL/机制？
- [ ] 是否询问用户确认核心工作？
- [ ] 排序是否按战略价值而非时间？
- [ ] 数字是否准确（44 not 8）？

### 9.4 不要遗漏关键工作

**案例**：SKILL_01a和SKILL_01b都是新增/完善的，不能只写一个

**检查方法**：
```bash
# 查找所有新增的SKILL文件
git log --since="..." --diff-filter=A --name-only | grep "SKILL"
# 查找所有修改的SKILL文件
git log --since="..." --name-only | grep "SKILL" | sort -u
```
