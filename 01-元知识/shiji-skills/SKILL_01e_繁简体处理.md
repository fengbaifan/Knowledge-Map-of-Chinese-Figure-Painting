# SKILL 01e: 繁简体处理

> **核心目标**：为《史记》提供准确的繁简体转换，特别是处理古籍中的特殊词汇和上下文相关转换。

---

## 一、背景与需求

### 1.0 为什么用算法转换而非维护两个版本

**核心设计决策**：项目采用"单一简体底本 + 算法转换"方案，而非维护繁简两套版本。

#### 原因1：中间分析过程无法维护双版本

项目的核心价值在于**深度分析**，而非仅提供阅读：

```
简体底本
  ↓
实体标注（人物、地点、时间、官职等）
  ↓
实体链接与知识图谱构建
  ↓
统计分析（词频、共现、关系网络）
  ↓
语义分析（事件提取、因果推理）
  ↓
拼音标注、校勘记录、注释系统
  ↓
最终HTML展示（简体/繁体切换）
```

**如果维护双版本**：
- ❌ 每个中间分析文件都需要双份（实体标注文件、校勘记录、统计数据等）
- ❌ 分析工具需要支持双语料输入
- ❌ 实体链接需要维护两套映射（"刘邦"/"劉邦" 都指向同一实体）
- ❌ 任何数据修正都要同步两份（工作量翻倍）

**采用算法转换**：
- ✅ 所有中间分析基于统一的简体底本
- ✅ 只在最终展示层转换繁体
- ✅ 分析流程无需关心繁简问题

#### 原因2：可扩展性需求

项目的长远目标不止《史记》：

**当前**：《史记》130篇，577,186字符
**未来**：扩展到数千部古籍，数千万字符

| 场景 | 维护双版本 | 算法转换 |
|-----|----------|---------|
| 《资治通鉴》(294卷) | 需人工校勘294卷繁体 | 算法自动处理，36条规则复用 |
| 《二十四史》(3000+卷) | 不可能完成 | 算法可行，逐步优化词表 |
| 新增1000部古籍 | 需1000×人工成本 | 同一算法，词表增量优化 |

**算法的优势**：
- ✅ **可复用**：36条规则可应用于所有古籍
- ✅ **可扩展**：发现新问题时增量添加规则
- ✅ **可验证**：统计驱动，每条规则有数据支撑
- ✅ **可维护**：规则集中管理，易于更新

#### 原因3：人工校勘的局限

**人工繁体校勘的问题**：
- ❌ **成本高**：577K字符需数月专业人工
- ❌ **不可复现**：每个人对异体字的选择不同
- ❌ **难以验证**：无法证明100%正确性
- ❌ **无法迭代**：底本更新后重新校勘

**算法转换的优势**：
- ✅ **成本低**：开发一次，多次使用
- ✅ **可复现**：相同输入必然相同输出
- ✅ **可验证**：统计分析183个错误，验证100%覆盖
- ✅ **可迭代**：底本更新后重新运行即可

#### 原因4：LLM分析的智能性需求

**为什么选择简体作为底本**：

现代大语言模型（LLM）主要基于简体中文训练：

| 因素 | 简体底本 | 繁体底本 |
|-----|---------|---------|
| **LLM训练语料** | 主要是简体（互联网内容80%+简体） | 繁体语料较少 |
| **实体识别准确率** | 高（"刘邦"、"项羽"训练充分） | 较低（"劉邦"、"項羽"训练不足） |
| **语义理解能力** | 强（简体是主流训练语料） | 较弱（繁体非主流） |
| **关系抽取效果** | 好（人名、地名、事件识别准确） | 较差（可能误识别） |

**具体影响**：

使用简体底本进行LLM分析时：
- ✅ **实体标注更准确**："刘邦"比"劉邦"更容易被正确识别
- ✅ **关系提取更可靠**：LLM对"汉高祖刘邦"的理解优于"漢高祖劉邦"
- ✅ **事件抽取更完整**：简体文本的语义解析更接近训练分布
- ✅ **知识图谱构建更智能**：实体链接、关系推理都受益于简体

**未来工作**：
- 项目计划使用LLM进行**逐章反思分析**，穷尽所有OpenCC未覆盖的错误
- 参见：[`skills/references/SKILL-01e-rules-繁简词表构造规则.md`](references/SKILL-01e-rules-繁简词表构造规则.md)
- 这个工作**必须基于简体底本**，因为LLM对简体的理解能力远超繁体

#### 总结

```
单一简体底本
    ↓
【大量中间分析】← 无需双版本维护
    ↓
算法转换（OpenCC + 36条规则）
    ↓
繁体展示（客户端切换）
```

这个方案：
- 支持**复杂的多层分析**（实体、关系、统计）
- 支持**大规模扩展**（数千部古籍）
- 避免**人工校勘瓶颈**（不可能为每部古籍手动校勘繁体）

### 1.1 繁简转换的挑战

古籍文本的繁简转换不能简单依赖标准转换工具（如OpenCC），因为：

1. **一字多义**：同一个简体字在不同语境下对应不同繁体字
   - 例：`后` → `後`（时间）vs `后`（君主配偶）
   - 例：`历` → `歷`（经历）vs `曆`（历法）
   - 例：`于` → `於`（介词）vs `于`（姓氏）

2. **专有名词**：人名、地名有特定的繁体写法
   - 例：`后稷` → `後稷`（人名，用"後"不用"后"）
   - 例：`于定国` → `于定国`（姓氏应保持"于"，不转"於"）

3. **古字用法**：古籍中保留的古字形不应转换
   - 例：`於` 不转为现代简化的 `于`

### 1.2 解决方案

采用**两层转换机制**：

1. **基础层**：使用 OpenCC (s2t) 进行标准简→繁转换
2. **校正层**：使用自定义词表 ([`s2t-custom-variants.json`](../docs/data/s2t-custom-variants.json)) 修正特殊词汇

**词表来源**：
- 通过统计分析《史记》577,186字符，识别87个一对多映射字符
- 分析OpenCC的183个实际错误案例
- 构建36条精准词组规则（无单字规则）

### 2.3 转换器实现

[`docs/js/simp-trad-converter.js`](../docs/js/simp-trad-converter.js) 核心功能：

#### 2.3.1 基础转换（OpenCC.js）

```javascript
// 使用 OpenCC.js 的 s2t (Simplified to Traditional) 转换
converter = OpenCC.Converter({ from: 'cn', to: 'tw' });
let result = converter(text);
```

#### 2.3.2 自定义词表校正

```javascript
// 加载自定义词表
const response = await fetch('../data/s2t-custom-variants.json');
customVariants = await response.json();

// 应用词表规则
for (const [simp, trad] of Object.entries(customVariants)) {
    const regex = new RegExp(escapeRegExp(simp), 'g');
    result = result.replace(regex, trad);
}
```

**重要**：按词汇长度降序替换，优先匹配长词避免误替换。

#### 2.3.3 DOM节点递归转换

```javascript
function convertNode(node, toTraditional) {
    // 跳过 <script>, <style>, <code> 等标签
    // 跳过段落编号（.para-num）
    // 只转换文本节点
    if (node.nodeType === Node.TEXT_NODE) {
        if (toTraditional) {
            originalTextCache.set(node, node.textContent); // 缓存原文
            node.textContent = convertText(node.textContent);
        } else {
            node.textContent = originalTextCache.get(node); // 恢复简体
        }
    }
    // 递归处理子节点
}
```

**优化**：
- 缓存原始文本，切回简体时无需反向转换
- 性能监控（`performance.now()`）

## 三、自定义词表构造（核心方法论）

### 3.1 问题定义

为OpenCC转换构建自定义词表，修正**上下文相关**的繁简转换错误。

**典型问题**：
- `"后"` → OpenCC统一转为`"後"`，但爵位应保持`"后"`
  - ✅ `"吕后"` → `"呂后"`（皇后，保持"后"）
  - ✅ `"后世"` → `"後世"`（时间，转为"後"）

**核心挑战**：如何区分以下三类差异？

| 差异类型 | 示例 | 是否应加入词表 |
|---------|------|--------------|
| **繁简转换差异** | `"后世"` OpenCC错转为`"后世"` | ✅ 应加入 |
| **异体字选择差异** | `"击"` OpenCC转`"擊"`，维基用`"撃"` | ❌ 不应加入 |
| **文本版本差异** | `"九年"` vs `"一年"` | ❌ 不应加入 |

### 3.2 五次尝试的完整历程

#### 尝试1：自动逐字比对（失败）

**方法**：
```python
for chapter in chapters:
    opencc_result = opencc.convert(simplified_text)
    wikisource_text = fetch_wikisource(chapter)
    diff = char_diff(opencc_result, wikisource_text)
    variants.append(diff)
```

**结果**：提取到8537条"差异"，其中99%是文本版本差异：
```
"九" → "一"         # 年份不同
"君" → "臣"         # 人物称谓不同
"盲，而舜母死" → "更娶妻而生象"  # 整句不同
```

**教训**：**不同版本的古籍不能直接逐字比对来提取繁简规则**。维基文库底本与项目底本不是同一版本，提取到的是**校勘差异**，不是繁简差异。

#### 尝试2：手动规则构造（成功 → v1.0/v2.0）

**方法**：
1. 识别已知问题（如"后"字问题）
2. 列举常见词汇（基于《史记》文本）
3. 构建精简规则集

**成果 v1.0**：32条精准规则

```json
{
  "吕后": "呂后",    // 爵位，保持"后"
  "太后": "太后",
  "后世": "後世",    // 时间，转为"後"
  "其后": "其後",
  "即": "卽",        // 古籍常用"卽"
  "原": "願",        // "原"作"愿"时
  "于": "於",        // 介词（通用规则）
  "琅": "瑯"         // 地名
}
```

**v2.0改进**：新增5条"发"、"历"字规则（37条）

**优点**：
- ✅ **精准**：每条规则都解决实际问题
- ✅ **可维护**：规则少，易理解
- ✅ **可验证**：每条规则在《史记》中有实例

**问题**：
- ⚠️ 包含5个单字规则（`"于": "於"`等），导致过度匹配
- ⚠️ `"于": "於"` 导致姓氏"于定国"错转为"於定國"
- ⚠️ `"即": "卽"`、`"原": "願"` 等OpenCC已正确处理，冗余

#### 尝试3：模糊匹配提取（部分成功）

**动机**：通过改进算法，**自动**从维基文库提取更多规则。

**方法**：
```python
from difflib import SequenceMatcher

def extract_variants_fuzzy():
    # 模糊对齐
    matcher = SequenceMatcher(None, opencc_text, wiki_text)
    for tag, i1, i2, j1, j2 in matcher.get_opcodes():
        if tag == 'replace':
            diff_pair = (opencc_text[i1:i2], wiki_text[j1:j2])
            if is_likely_variant_not_version(diff_pair):
                variants[diff_pair] += 1

    # 频率过滤（≥3次）
    return {k: v for k, v in variants.items() if freq >= 3}
```

**结果**：提取405条规则，筛选出48条高频规则（≥3次）

**验证结果**：
- **23条引号规则**：标点风格差异（大陆`""`vs台湾`「」`）
- **22条异体字规则**：OpenCC已正确转换，只是异体字选择不同
  - `"击"` → OpenCC用`"擊"` ✅，维基用`"撃"`（异体字选择差异）
  - `"录"` → OpenCC用`"彔"` ✅，维基用`"錄"`（异体字选择差异）
- **3条重复规则**：与手动词表重复
- **0条有效新规则**

**关键发现**：**无法区分"繁简转换差异"和"异体字选择差异"**

**教训**：
1. 与维基文库比较不适合构建繁简词表
2. **异体字选择差异 ≠ OpenCC缺陷**
3. 自动提取方法需要先验证OpenCC是否已正确处理

**成功之处**：
- ✅ 验证了手动规则的高频性
- ✅ 模糊匹配成功过滤了版本差异（相比尝试1的8537条噪声）
- ✅ 识别出了标点风格差异问题

#### 尝试4：统计分析+精确错误识别（最终成功 → v3.0）

**方法**：
1. **全文统计分析**（[`scripts/analyze_simp_trad_mapping.py`](../scripts/analyze_simp_trad_mapping.py)）
   - 分析《史记》577,186字符
   - 识别87个一对多映射字符（占1.8%字符种类，但出现19,493次，占3.38%文本）
   - 分类：相同（70.1%）、一对一（28.1%）、一对多（1.8%）

2. **精确错误定位**（[`scripts/find_actual_errors.py`](../scripts/find_actual_errors.py)）
   - 只统计**真实错误**（OpenCC转换不符合《史记》用法）
   - 区分"姓氏于"（应保持"于"）vs"介词于"（应转"於"）
   - 区分"头发"（应转"髮"）vs"出发"（应转"發"）
   - 区分"历法"（应转"曆"）vs"经历"（应转"歷"）

3. **结果**：识别出183个真实错误
   - "于"字姓氏：134个错误（于定国、于单等）
   - "发"字头发义：10个错误（发上指等）
   - "历"字历法义：39个错误（历日、历数等）

4. **v3.0词表生成**（[`scripts/create_v3_final.py`](../scripts/create_v3_final.py)）
   - **移除5条单字规则**（过度匹配）
   - **新增9条精准词组规则**
   - **最终：36条规则，全部为2字及以上词组**

**v3.0特点**：
- ✅ **无单字规则**：避免过度匹配
- ✅ **100%覆盖率**：修复所有183个真实错误
- ✅ **文件重命名**：`custom-variants.json` → `s2t-custom-variants.json`（明确用途）

#### 尝试5：词频统计+OpenCC验证（正在进行 → v4.0）

**日期**：2026-04-01

**动机**：v3.0虽然修复了183个错误，但覆盖面仍有限（仅36条规则）。需要系统性地对**全部130章的所有词汇**进行完整覆盖。

**方法**：

1. **全文分词与词频统计**（[`scripts/build_complete_word_variants.py`](../scripts/build_complete_word_variants.py)）
   ```
   步骤1：对全部130章进行分词（jieba）
   步骤2：统计所有词的出现频率
   步骤3：对出现≥1次的词，提取其繁体形式
          （方法：与corpus/shiji/史記正文.繁体.txt对照）
   步骤4：筛选出繁体≠简体的词
   ```

2. **结果**：
   - 总词数（含重复）：190,119词
   - 唯一词数：65,916词
   - 成功映射：38,884词
   - **繁简不同（需要转换）：10,028词**

3. **OpenCC转换验证**（[`scripts/validate_opencc_conversion.py`](../scripts/validate_opencc_conversion.py)）
   ```python
   对10,028个词逐一验证：
   - 用OpenCC（s2t）转换简体词
   - 对比OpenCC结果与期望繁体
   - 记录转换错误的词条
   ```

4. **验证结果**：
   - **转换正确**：8,865词（88%）
   - **转换错误**：1,163词（11%）⚠️

   **错误类型分布**：

   | 错误类型 | 数量 | 占比 | 示例 |
   |---------|------|------|------|
   | **為→爲** | 983词 | 84.5% | 以为→期望"以為"，OpenCC给"以爲" |
   | **其他** | 111词 | 9.5% | 齐栗、游孙、扑师武等 |
   | **于→於** | 38词 | 3.3% | 迁于→期望"遷于"，OpenCC给"遷於" |
   | **辟→闢** | 15词 | 1.3% | 尧辟位→期望"堯辟位"，OpenCC给"堯闢位" |
   | **咸→鹹** | 4词 | 0.3% | 咸归→期望"咸歸"，OpenCC给"鹹歸"（误转为食盐） |
   | **采→採** | 4词 | 0.3% | 采有国→期望"采有國"，OpenCC给"採有國" |
   | **余→餘** | 4词 | 0.3% | 余欲观→期望"余欲觀"，OpenCC给"餘欲觀" |
   | **干→乾/幹** | 4词 | 0.3% | 杨干→期望"楊干"，OpenCC给"楊乾/楊幹" |

**关键发现**：

1. **"為"字异体问题是主要问题**（84.5%）
   - OpenCC将"为"转为"爲"，但维基文库使用"為"
   - 这是OpenCC默认配置与维基文库编辑偏好的差异
   - 涉及983个词，如"以为"、"为帝"、"最为"等

2. **"于"字转换问题**（3.3%）
   - OpenCC倾向将"于"转为"於"
   - 在人名/地名中应保持"于"
   - 例："迁于"应为"遷于"而非"遷於"

3. **专有名词误转**（1.9%）
   - "咸"（都、皆）误转为"鹹"（食盐）
   - "采"（采邑）误转为"採"（采摘）

### 3.4 核心认识

#### 3.4.1 三类差异的本质区别

| 差异类型 | 定义 | 示例 | OpenCC是否正确 | 是否加入词表 |
|---------|------|------|--------------|------------|
| **繁简转换差异** | OpenCC转换结果不符合语境 | `"后世"` 应转`"後世"`，OpenCC错转为`"后世"` | ❌ 错误 | ✅ 应加入 |
| **异体字选择差异** | OpenCC转换正确，但异体字选择不同 | `"击"` → OpenCC用`"擊"`，维基用`"撃"` | ✅ 正确 | ❌ 不加入 |
| **文本版本差异** | 底本不同导致的文字差异 | `"九年"` vs `"一年"` | - | ❌ 不加入 |

#### 3.4.2 词表构造原则（最终版）

1. ✅ **只添加OpenCC未正确转换的规则**
   - 必须先验证：`opencc.convert(简体) ≠ 期望繁体`

2. ✅ **只使用词组规则，禁止单字规则**
   - 最短规则：2字
   - 最长规则：3字
   - 避免过度匹配

3. ❌ **不追求与维基文库完全一致**
   - 异体字选择是编辑偏好，不是转换问题

4. ✅ **聚焦核心问题**
   - 上下文相关的字符转换（如"后"字）
   - 专有名词（如姓氏"于定国"）

5. ✅ **统计驱动，精准修复**
   - 基于全文577,186字符的统计分析
   - 识别真实错误（183个）而非噪声

#### 3.4.3 方法论边界

| 方法 | 适用场景 | 不适用场景 |
|-----|---------|-----------|
| **逐字比对** | 同一版本的繁简对照 | ❌ 不同版本的古籍 |
| **模糊匹配** | 过滤版本差异、识别趋势 | ❌ 区分转换差异和异体字差异 |
| **手动构建** | ✅ 解决已知核心问题 | 大规模覆盖（成本高） |
| **统计分析+精确错误定位** | ✅ 构建高精度词表 | 需要完整文本语料 |
| **与维基比较** | 发现潜在问题 | ❌ 直接作为词表来源 |

### 4.2 验证现有规则

```python
# 验证s2t-custom-variants.json中的规则是否必要
from opencc import OpenCC
import json

cc = OpenCC('s2t')
with open('docs/data/s2t-custom-variants.json') as f:
    variants = json.load(f)

for simp, trad in variants.items():
    opencc_result = cc.convert(simp)
    if opencc_result == trad:
        print(f"⚠️ 规则冗余：{simp} → {trad}")
        print(f"   OpenCC已正确转换")
    else:
        print(f"✅ 规则必要：{simp}")
        print(f"   OpenCC: {opencc_result} ≠ 期望: {trad}")
```

### 10.2 成功的做法

| 尝试 | 方法 | 规则数/数据量 | 成功原因 |
|-----|------|--------------|---------|
| 尝试2 | 手动构建 | 32条 | 聚焦核心问题，精准解决 |

### 10.3 四个关键认识

1. **文本版本差异** ≠ 繁简转换差异（尝试1的教训）
2. **异体字选择差异** ≠ OpenCC缺陷（尝试3的教训）
3. **维基文库不适合作为繁简词库的构建来源**
4. **"為/爲"异体字差异是主要问题**（尝试5的发现）
   - OpenCC与维基文库在"為/爲"选择上有系统性差异
   - 占所有转换错误的84.5%（983/1,163）
   - 需要判断是否作为"编辑偏好"而接受

### 10.4 方法论启示

对于古籍繁简词库构建：
- ✅ **统计驱动**：基于全文统计分析，识别真实问题
- ✅ **精确定位**：区分"繁简错误"、"异体字差异"、"版本差异"
- ✅ **精简优先**：36条精准规则 > 数百条含噪声的规则
- ✅ **验证驱动**：每条规则都需要OpenCC验证
- ✅ **只用词组**：禁止单字规则，避免过度匹配
- ❌ **避免自动化陷阱**：自动提取需要非常谨慎的验证
