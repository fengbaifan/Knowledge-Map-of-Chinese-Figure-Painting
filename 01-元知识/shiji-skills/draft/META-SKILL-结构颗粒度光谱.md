# 元技能14: 结构颗粒度光谱 — 任务驱动的知识表示选择


---

**说明**: 本文中的成本估算($1, $5, $50等)并非真实货币投入,而是作为**相对计算量单位**,可理解为token消耗或人工工作量的相对比例。实际成本取决于具体实现方式(Agent自动化程度、人工审查比例等)。

**本项目实践**: 史记知识库在[kg/ontology/ontology-v1/](../../kg/ontology/ontology-v1/)中实现了不同颗粒度的SKU(标准知识单元),包括434项Factual SKUs、241项Procedural SKUs和结构化关系知识,是本文理论的实践验证。

---

## 一、核心思想

> **"不是所有知识都需要最高程度的结构化,任务决定颗粒度"**

**结构颗粒度光谱**（Structural Granularity Spectrum）是指从完全无结构的自然语言文本到完全结构化的知识图谱之间的连续表示空间,不同的任务需要不同颗粒度的知识结构。

### 核心洞察

```
错误观念: "知识图谱就是要把所有信息都结构化"

正确认识:
  - 全量结构化成本高昂(10-100倍于文本)
  - 许多任务并不需要完全结构化
  - 最优策略是"任务适配的最小结构化"

关键公式:
  结构化投入 ∝ 颗粒度²
  任务收益 ∝ f(颗粒度) [非线性,有拐点]

最优颗粒度: 边际收益 = 边际成本
```

## 三、六层颗粒度详解

### L0: 原始文本（无结构）

#### 特征

```
表现形式:
  连续的自然语言文本,无任何标注或结构化

示例:
  黄帝者少典之子姓公孙名曰轩辕生而神灵弱而能言幼而徇齐长而敦敏成而聪明...

优点:
  - 构建成本:零
  - 保留完整信息
  - 人类可读性最强

缺点:
  - 查询需全文扫描
  - 无法精确检索
  - 推理能力弱
```

---

#### 适用任务

| 任务类型 | 示例 | 说明 |
|---------|------|------|
| **全文阅读** | "阅读《史记》全文" | 人类阅读,无需结构化 |
| **模糊搜索** | "查找包含'黄帝'的段落" | Grep/全文检索即可 |
| **美学欣赏** | "品味太史公文笔" | 结构化会破坏美感 |

**成本**: 1x (无额外投入,基准)

---

### L1: 向量化（密集向量表示）

#### 特征

```
表现形式:
  文本 → Embedding向量(通常768维或1536维)

技术:
  - text-embedding-ada-002 (OpenAI)
  - BGE-large (智源)
  - voyage-multilingual-2 (Voyage AI)

数据结构:
  paragraph_001: [0.023, -0.187, 0.451, ..., 0.092]  # 768维
  paragraph_002: [0.102, 0.034, -0.289, ..., 0.217]

检索方式:
  query → embedding_query
  相似度 = cosine_similarity(embedding_query, embedding_doc)
  返回 top_k 最相似文档
```

---

#### 优点 vs 缺点

**优点**:
- 构建成本低(自动Embedding,无需人工)
- 支持语义相似度搜索(而非仅关键词)
- 多模态融合(文本/图像/音频)

**缺点**:
- 黑箱表示(无法解释为何相似)
- 召回不精确(相似≠相关)
- 无结构约束(可能召回矛盾信息)

---

#### 适用任务

| 任务类型 | 示例 | 向量化方案 |
|---------|------|-----------|
| **语义搜索** | "查找与'商鞅变法'相似的改革" | query+doc embedding,cosine |
| **文档聚类** | "将130篇按主题分组" | k-means on embeddings |
| **相似推荐** | "读了《商君列传》,推荐类似篇章" | 计算embedding距离 |

**成本**: 2x (Embedding计算成本)

---

### L2: 轻量标注（Lightweight Markup）

#### 特征

```
表现形式:
  文本 + 浅层实体标注

示例:
  〖@黄帝〗者,〖@少典〗之子,姓〖&公孙〗,名曰〖@轩辕〗。
  生而神灵,弱而能言...

标注类型:
  - 18类实体(人物@、地名=、官职%、邦国&...)
  - 保留原始文本
  - 仅边界标记,无深层语义

数据结构:
  {
    "text": "黄帝者,少典之子,姓公孙,名曰轩辕。",
    "entities": [
      {"type": "person", "name": "黄帝", "span": [0, 2]},
      {"type": "person", "name": "少典", "span": [3, 5]},
      {"type": "surname", "name": "公孙", "span": [8, 10]},
      {"type": "person", "name": "轩辕", "span": [13, 15]}
    ]
  }
```

---

#### 优点 vs 缺点

**优点**:
- 构建成本中等(Agent辅助,半自动)
- 保留文本完整性(渲染时可还原)
- 支持实体检索(grep `〖@刘邦〗`)
- 人类可读(标注符号不破坏阅读)

**缺点**:
- 无关系信息(父子/君臣需推断)
- 无时间信息(事件顺序不明)
- 歧义未消除("武王"可能是多人)

---

#### 适用任务

| 任务类型 | 示例 | 操作方式 |
|---------|------|---------|
| **实体检索** | "查找所有提到'刘邦'的段落" | `grep "〖@刘邦〗" *.md` |
| **实体统计** | "统计史记中有多少个人物" | 提取所有`〖@...〗`,去重 |
| **上下文阅读** | "显示'商鞅'周围的原文" | 定位标注,提取±200字 |
| **HTML渲染** | "人物名可点击跳转" | `〖@name〗` → `<a href="#">name</a>` |

**成本**: 5x (Agent标注 + 人工审查)

---

### L3: 半结构化（Hybrid: Core Graph + Text Annotations）

#### 特征

```
表现形式:
  核心关系结构化 + 自然语言注释

架构:
  实体 (结构化)
    ├─ 核心属性 (JSON)
    └─ 富文本描述 (Markdown)

  事件 (结构化索引 + 原文片段)
    ├─ 元数据: {id, type, year, persons, locations}
    └─ 描述: "秦使左庶长王龁攻韩,取上党..."

示例:
  {
    "entity_id": "刘邦",
    "type": "person",
    "core_facts": {
      "aliases": ["沛公", "汉王", "高祖"],
      "birth_year": -256,
      "death_year": -195
    },
    "annotations": [
      {
        "source": "008-para-003",
        "text": "高祖,沛丰邑中阳里人,姓刘氏,字季。"
      },
      {
        "source": "008-para-045",
        "text": "汉王至荥阳,诸败军皆会,萧何亦发关中老弱未傅悉诣荥阳..."
      }
    ]
  }
```

---

#### 核心设计原则

**结构化部分**:
- 关键事实(生卒年、别名、身份)
- 关键关系(父子、君臣、战争双方)
- 时空坐标(年份、地点)

**文本注释部分**:
- 复杂事件描述(保留原文,避免信息损失)
- 评价性内容("太史公曰")
- 歧义性描述(多源冲突,保留所有版本)

**分界线判断**:
```
结构化: 能用三元组清晰表达 (刘邦, 父亲, 刘太公)
文本注释: 需要长句才能说清 ("高祖为人,隆准而龙颜,美须髯...")
```

---

#### 优点 vs 缺点

**优点**:
- 精确查询(结构化部分)
- 信息完整(文本注释保留细节)
- 成本可控(仅核心结构化)
- 灵活扩展(文本注释易补充)

**缺点**:
- 边界模糊(哪些该结构化?)
- 查询复杂度高(结构+文本混合检索)
- 一致性维护难(结构与注释可能不一致)

---

#### 适用任务

| 任务类型 | 示例 | 查询方式 |
|---------|------|---------|
| **实体链接** | "沛公指谁?" | 查询`aliases`字段 → 刘邦 |
| **时间推理** | "刘邦即位时多少岁?" | (-195) - (-256) = 61岁 |
| **关系搜索** | "刘邦的父亲是谁?" | 查询`relations[type=父子]` |
| **证据检索** | "刘邦攻荥阳的原文在哪?" | 搜索`annotations[text含荥阳]` |

**成本**: 20x (核心事实提取 + 关系标注)

---

### L4: 核心图谱（Core KG + Text Fallback）

#### 特征

```
表现形式:
  骨架知识图谱 + 原文文本兜底

架构:
  三元组 (80%知识)
    (刘邦, 父亲, 刘太公)
    (刘邦, 配偶, 吕雉)
    (刘邦, 卒于, -195)
    (刘邦, 出生地, 沛县)

  长描述文本 (20%知识)
    description: "高祖为人,隆准而龙颜,美须髯,左股有七十二黑子..."

数据结构:
  {
    "nodes": [
      {"id": "刘邦", "type": "person", "birth": -256, "death": -195},
      {"id": "吕雉", "type": "person"},
      {"id": "沛县", "type": "place"}
    ],
    "edges": [
      {"source": "刘邦", "relation": "父亲", "target": "刘太公"},
      {"source": "刘邦", "relation": "配偶", "target": "吕雉"},
      {"source": "刘邦", "relation": "出生地", "target": "沛县"}
    ],
    "text_annotations": {
      "刘邦": {
        "appearance": "高祖为人,隆准而龙颜,美须髯,左股有七十二黑子",
        "personality": "仁而爱人,喜施,意豁如也",
        "评价": "太史公曰:夏之政忠,忠之敝,小人以野..."
      }
    }
  }
```

---

#### 核心图谱覆盖范围

**必须结构化**(关键事实):
1. **身份属性**: 生卒年、别名、身份、官职
2. **核心关系**: 父子、夫妻、君臣、师徒
3. **关键事件**: 战争、改革、崩逝、封赏
4. **时空坐标**: 事件年份、发生地点

**可选文本兜底**(复杂描述):
1. **外貌描述**: "隆准而龙颜,美须髯"
2. **性格评价**: "仁而爱人,意豁如也"
3. **复杂事件**: "萧何月下追韩信"(多步骤,难拆分)
4. **太史公曰**: 评论性内容

---

#### 优点 vs 缺点

**优点**:
- 高精度推理(图谱路径搜索)
- 关系明确(无需LLM推断)
- 一致性强(约束检查)
- 可视化强(图谱展示)

**缺点**:
- 构建成本高(需人工审查)
- 覆盖不完整(长尾知识缺失)
- 更新维护难(关系变更成本高)

---

#### 适用任务

| 任务类型 | 示例 | 操作方式 |
|---------|------|---------|
| **关系推理** | "刘邦和项羽的共同朋友" | 图谱路径搜索 |
| **家族树** | "刘邦的子孙后代" | BFS遍历`父子`关系 |
| **时间线** | "刘邦一生的关键事件" | 查询涉及刘邦的events,排序 |
| **冲突检测** | "刘邦-195年卒,但-190年有事件?" | 约束验证 |

**成本**: 50x (深度标注 + 关系抽取 + 一致性验证)

---

### L5: 全量知识图谱（Complete KG）

#### 特征

```
表现形式:
  所有信息完全结构化,无文本兜底

目标:
  - 所有实体结构化(5000+人物)
  - 所有关系显式化(父子/君臣/朋友/敌对/...)
  - 所有事件拆解(多步骤事件 → 原子事件序列)
  - 所有时空精确化(具体到年月日)
  - 所有评价量化(太史公评价 → 情感分数)

示例:
  原文: "萧何月下追韩信"

  拆解为原子事件:
    事件1: 韩信离开汉营 (year=-206, month=8, location=汉中)
    事件2: 萧何发现韩信离开 (同日)
    事件3: 萧何追赶韩信 (同日,夜间)
    事件4: 萧何劝说韩信返回 (次日)
    事件5: 韩信返回汉营 (次日)
    事件6: 刘邦拜韩信为大将军 (数日后)

  关系网络:
    (萧何, 推荐, 韩信) {time: -206, context: "追韩信事件"}
    (韩信, 任职, 大将军) {appointer: 刘邦, time: -206-08}
    (刘邦, 信任, 韩信) {degree: 0.95, source: "拜大将"}
```

---

#### 全量图谱的层次

**L5-1: 完整事实层**
- 每句话拆解为三元组
- 示例: "高祖为人,隆准而龙颜" → (刘邦, 鼻型, 隆准) + (刘邦, 额型, 龙颜)

**L5-2: 深层语义层**
- 隐含关系显式化
- 示例: "萧何月下追韩信" → (萧何, 看重, 韩信) + (韩信, 不满, 刘邦待遇) + (萧何, 担心, 韩信离开)

**L5-3: 评价量化层**
- 情感/态度数值化
- 示例: "太史公曰:韩信,国士无双" → (韩信, 太史公评价, 9.5/10)

**L5-4: 时空精确层**
- 所有事件具体到日
- 示例: "沛公起兵" → (刘邦, 起兵, -209-09-01)

---

#### 优点 vs 缺点

**优点**:
- 推理能力最强(可执行复杂SPARQL查询)
- 一致性最高(全局约束检查)
- 可解释性最强(所有推理路径可追溯)
- 多语言支持(概念层统一,标签层翻译)

**缺点**:
- 构建成本极高($100/千字,10倍于L4)
- 信息损失(拆解过程中丢失原文韵味)
- 维护成本高(更新一个事实可能影响数十个三元组)
- 过度结构化(某些模糊知识强行量化会失真)

---

#### 适用任务

| 任务类型 | 示例 | 说明 |
|---------|------|------|
| **复杂推理** | "谁是秦统一的关键人物?" | 多跳路径聚合 |
| **假设分析** | "如果商鞅不被车裂,秦会更早统一吗?" | 反事实推理 |
| **跨语言** | "Translate KG to English" | 概念层不变,标签翻译 |
| **一致性审计** | "检测所有时序矛盾" | 全局约束验证 |

**成本**: 100x (完全人工审查 + 深度标注)

## 四、任务-颗粒度匹配矩阵

### 4.1 常见任务的最优颗粒度

| 任务类型 | 最优颗粒度 | 次优颗粒度 | 说明 |
|---------|----------|----------|------|
| **全文阅读** | L0(原文) | L2(轻标注) | 结构化会破坏阅读体验 |
| **关键词搜索** | L0(grep) | L1(向量) | 简单关键词无需结构化 |
| **语义搜索** | L1(向量) | L2(标注) | 向量最适合相似度计算 |
| **问答系统** | L2(标注) | L3(半结构) | 实体边界+RAG检索 |
| **实体统计** | L2(标注) | L3(半结构) | 提取标注即可统计 |
| **关系查询** | L4(核心图谱) | L3(半结构) | 关系需显式结构化 |
| **家族树** | L4(核心图谱) | - | 必须有完整父子关系 |
| **时间推理** | L3(半结构) | L4(核心图谱) | 年份结构化即可 |
| **因果推理** | L4(核心图谱) | L5(全量) | 需要事件关系网络 |
| **一致性检查** | L5(全量) | L4(核心图谱) | 全局约束需完整图谱 |
| **多语言KG** | L5(全量) | - | 概念层需完全结构化 |

---

### 4.2 成本-收益分析

```
任务: "查找刘邦的所有敌人"

L0 (原文):
  方法: 人工阅读130篇,手动记录
  成本: 100小时 × $50 = $5,000
  准确率: 95% (可能遗漏)

L1 (向量):
  方法: Embedding搜索"刘邦的敌人"
  成本: $10 (API费用)
  准确率: 60% (召回不精确,"朋友"也可能被召回)

L2 (轻标注):
  方法: 检索〖@刘邦〗段落 + LLM提取敌对关系
  成本: $50 (检索 + LLM调用)
  准确率: 75% (需推理,可能误判)

L3 (半结构):
  方法: 查询events中涉及刘邦的战争,提取对手
  成本: $100 (事件索引已构建,直接查询)
  准确率: 85% (战争敌人完整,政治敌人需推断)

L4 (核心图谱):
  方法: 查询(刘邦, 敌对, ?)关系
  成本: $200 (图谱已构建,关系查询)
  准确率: 95% (关系显式,覆盖全面)

L5 (全量):
  方法: SPARQL复杂查询(敌对+战争+政治冲突)
```

## 六、颗粒度选择决策树

### 6.1 决策流程

```
问题: 我的任务应该选择哪个颗粒度?

步骤1: 任务类型判断
  ├─ 需要精确关系查询? (如"父子关系")
  │   └─ 是 → L4或L5
  │   └─ 否 → 继续
  │
  ├─ 需要时间推理? (如"事件年代排序")
  │   └─ 是 → L3或L4
  │   └─ 否 → 继续
  │
  ├─ 需要实体边界? (如"提取所有人名")
  │   └─ 是 → L2或L3
  │   └─ 否 → 继续
  │
  ├─ 需要语义相似? (如"查找类似文章")
  │   └─ 是 → L1
  │   └─ 否 → L0(原文阅读)

步骤2: 成本预算评估
  ├─ 预算充足(>$50/千字) → 可选L4-L5
  ├─ 预算中等($10-50/千字) → 推荐L3
  ├─ 预算有限(<$10/千字) → 推荐L1-L2
  └─ 无预算 → L0(原文)

步骤3: 时间窗口检查
  ├─ 需要立即上线(1周内) → L1(自动Embedding)
  ├─ 可等待1个月 → L2-L3(半自动)
  ├─ 长期项目(3-12个月) → L4-L5
  └─ 持续演进 → 从L1开始,逐步升级

步骤4: 数据规模考虑
  ├─ 数据量巨大(>1000万字) → L1-L2(自动化)
  ├─ 数据量中等(10-100万字) → L2-L3
  └─ 数据量小(<10万字) → 可尝试L4-L5

最终决策: 取上述四步的交集
```

- L0 → L2: $7

### 7.2 降级成本(从高到低)

| 转换路径 | 成本 | 主要工作 | 质量损失 |
|---------|-----|---------|---------|
| **L5 → L4** | 几乎免费 | 隐藏部分关系 | 5% (长尾知识) |

**关键洞察**:
- 升级成本高昂(增量投入)
- 降级几乎免费(信息丢弃)
- 但降级不可逆(结构化知识难以重建)

**策略建议**:
- 谨慎选择初始颗粒度(降级容易,升级难)
- 优先考虑长期需求(避免后期推倒重来)
- 模块化构建(不同模块不同颗粒度)

### 8.2 查询路由策略

```python
def route_query(query, query_type):
    """
    根据查询类型路由到合适的颗粒度层
    """

    if query_type == "full_text_search":
        # 关键词搜索 → L0层
        return grep_search(query, chapter_files)

    elif query_type == "semantic_search":
        # 语义搜索 → L1层
        query_embedding = embed(query)
        return vector_search(query_embedding, embeddings_db)

    elif query_type == "entity_extraction":
        # 实体提取 → L2层
        return extract_entities_from_tagged_files(query)

    elif query_type == "event_query":
        # 事件查询 → L3层
        return query_event_index(query, event_index_files)

    elif query_type == "relation_query":
        # 关系查询 → L4层
        return graph_search(query, knowledge_graph)

    elif query_type == "complex_reasoning":
        # 复杂推理 → L5层(如果存在)
        if full_graph_exists():
            return sparql_query(query, full_graph)
        else:
            # 降级到L4
            return graph_search_with_llm_fallback(query, kg_l4)

    else:
        # 默认:混合策略
        return hybrid_search(query, all_layers)
```

```python
    混合颗粒度查询

    wars = query_event_index(

            'belligerents.attacker.state': '秦',

        }

    key_persons = set()

        key_persons.update(commanders)

    # Step 3: L3层 - 查询人物详情(生卒年/身份)

        person_details.append({

            'birth': details['birth_year'],
            'death': details['death_year'],
            'role': details.get('role')
```

## 九、常见误区与最佳实践

### 9.1 误区1:"知识图谱就是要全量结构化"

**错误观念**:
> "既然要做知识图谱,就要把所有信息都变成三元组"

**正确认识**:
- L5全量图谱成本极高($100/千字)
- 大部分任务L3-L4即可满足
- 过度结构化会丢失原文美感(如太史公曰)

**最佳实践**:
- 核心事实结构化(L3-L4)
- 复杂描述保留文本(L2-L3)
- 混合架构:图谱+文本注释

---

### 9.2 误区2:"向量化可以替代结构化"

**错误观念**:
> "有了Embedding,就不需要知识图谱了"

**正确认识**:
- 向量化擅长:相似度搜索
- 向量化不擅长:精确关系查询、时间推理、一致性验证

**对比**:
```
任务: "刘邦的父亲是谁?"

向量方法:
  Embedding("刘邦的父亲") → 召回top-3段落
  → 可能召回: "刘邦者,沛人也"(相似但无答案)
  → 需要LLM二次推理

图谱方法:
  query_relation(subject="刘邦", relation="父亲")
  → 直接返回: "刘太公"
  → O(1)查询,100%准确
```

**最佳实践**:
- L1(向量) + L3(半结构)混合
- 向量用于召回,结构用于精确答案

---

### 9.3 误区3:"一开始就选最高颗粒度"

**错误观念**:
> "反正最后要L5,不如一开始就直接做L5"

**问题**:
- L5构建周期长(1年+)
- 需求可能变化(白费投入)
- 错过MVP上线时机

**最佳实践**:
- **MVP优先**: 从L1-L2开始,快速验证需求
- **递进式构建**: L1 → L2 → L3,每个阶段交付价值
- **按需升级**: 仅在确认需求后才投入L4-L5

**案例**:
```
史记项目演进路线:
  第1个月: L0(原文) + L1(Embedding) → 上线阅读功能
  第2-3个月: L2(轻标注) → 实体检索、HTML渲染
  第4-6个月: L3(事件索引+编年表) → 时间推理、事件查询
  第7-12个月: L4(关系图谱) → 关系查询、可视化
  第2年(可选): L5(全量图谱) → 学术合作、多语言

每个阶段都有可交付产品,而非等1年后才上线
```

---

### 9.4 误区4:"不同模块必须用同一颗粒度"

**错误观念**:
> "项目要统一标准,所有章节都用L4"

**问题**:
- 不同内容适合不同颗粒度
- 强制统一会浪费资源

**最佳实践**:
- **核心内容高颗粒度**: 本纪/世家 → L3-L4
- **次要内容低颗粒度**: 书/表 → L2
- **评论性内容保留原文**: 太史公曰 → L0+L2

## 十、总结

### 核心要点

1. **颗粒度光谱**:L0(原文) → L1(向量) → L2(轻标注) → L3(半结构) → L4(核心图谱) → L5(全量)
2. **任务决定颗粒度**:不是越高越好,而是"刚好够用"
3. **成本-收益权衡**:边际收益 = 边际成本的拐点是最优选择
4. **递进式构建**:从低到高,每个阶段交付价值
5. **混合架构**:不同模块不同颗粒度,查询时动态路由

### 决策原则

**Occam's Razor for KG**:
> "在满足任务需求的前提下,选择最低的结构化颗粒度"

**理由**:
- 降低构建成本
- 缩短交付周期
- 保留升级空间
- 避免过度工程

### 与其他元技能的关系

- **← 柳叶刀方法**:分解任务 → 确定所需颗粒度
- **← 上下文压缩**:结构化是压缩的一种手段
- **← 数据融合**:融合结果的颗粒度需统一
- **← 成本控制**:颗粒度直接影响成本
- **→ 技能迁移**:L3-L4更易迁移(规律可复用)

**结构颗粒度是知识工程的"调节旋钮"**:拧得太松(L0-L1),查询能力弱;拧得太紧(L5),成本爆炸。找到适合任务的刻度,才是工程智慧。
