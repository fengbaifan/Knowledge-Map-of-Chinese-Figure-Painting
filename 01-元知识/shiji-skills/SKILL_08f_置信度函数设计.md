# SKILL 08f: 置信度函数设计

> 把「这条判断有多可信」变成可计算、可比较、可分流的数。
>
> 核心立场：**置信度不是先验设计出来的，是从反思样本里归纳出来的。** 每加一个信号、每调一个阈值，都应能指回某次反思中出现的一类证据。

---

## 一、何时使用

满足以下任一条件，即应为该工序设计置信度函数：

- 输出量大（>500 条），人工不可能逐条复核，需要按可信度排序
- 输出会被下游消费（融合/推理/渲染），下游需要知道「这条能不能信」
- 同一项判断可来自多个信号源，需要决定**何时采纳、何时弃权、何时多标签**
- 已经积累了一轮以上反思报告，能从中识别出反复出现的证据模式

不需要置信度的工序：判断本身就是 0/1 确定的（如格式 lint）、样本量极小可全人工。

---

## 二、核心方法：反思归纳式

整个流程是 **样本 → 反思 → 信号 → 分层规则 → 打分函数** 的循环，**不要反过来**。

| 阶段 | 做什么 | 工件 |
|---|---|---|
| 1. 跑一遍朴素方法 | 用最简单的 baseline（白名单 / 后缀 / 全库统计）出一批结果 | 初版输出 JSON |
| 2. 抽样反思 | 人工或 Agent 审 50-200 条，标注「对/错/存疑」并写出**判错的依据** | reflection.md |
| 3. 归纳信号 | 把判错依据合并成可机读的特征：上下文窗口词、共现实体、来源条数、注释命中… | 规则草稿 |
| 4. 分层落地 | 信号按强度排成 L1→Ln 层，强信号先决断，弱信号兜底 | 脚本中的分层函数 |
| 5. 标定阈值 | 在已审样本上回测，决定 majority 比例（常用 0.6）、最小命中数（常用 1-2）、保守边界 | 参数注释里写明回测来源 |
| 6. 标记不确定 | 函数对低置信样本**主动弃权或多标签**，留给下一轮反思 | 输出中的"未分类"/"多标签" |
| 7. 迭代 | 下游或人工再反思一批，回到步骤 3 | 累积反思工件 |

**反例（要避免）**：先拍脑袋写一个加权公式（如 `0.4*A + 0.3*B + 0.3*C`），再去对数据。这种公式调不动也解释不了，等价于赌博。

## 三、四种通用模式

### 模式 A：多源票数法

多个独立信号各投一票，按 `Counter()` 汇总，主导项占比 ≥ 阈值时采纳。

- 用于：分类、消歧、打标签
- 关键参数：**majority**（常用 0.6）、**min_hits**（常用 1-2）
- 实例：[`kg/entities/scripts/classify_places.py`](../kg/entities/scripts/classify_places.py) 的 L3 共现词规则
- 实例：[`scripts/infer_single_char_names.py`](../scripts/infer_single_char_names.py) 的同章共现全名计数

### 模式 B：分层兜底（优先级递推）

证据强度递减的 L1→Ln 层。每层只在前面所有层都未命中时启用；强类别一旦命中，弱规则不得覆盖。

- 实例：[`SKILL_03h_地名地段分类.md`](SKILL_03h_地名地段分类.md) 的 L1-L5（白名单 → 章节上下文 → 共现 → 注释 → 并列传播）
- 实例：[`SKILL_03b_人名消歧.md`](SKILL_03b_人名消歧.md) 的四级优先级（前缀国名 → 邻近国名 → 章节主题 → 全名共现）

### 模式 C：三级不确定度标记

不要硬塞 0-1 浮点；用三级（精确 / 推算 / 近似）就够下游使用。

- 事件年代规范：`(公元前X年)` 精确 / `[公元前X年]` 确定推算 / `[约公元前X年]` 近似 — 见 [`SKILL_04c_事件年代推断.md`](SKILL_04c_事件年代推断.md)
- 人物生卒年：无前缀 = 精确、`约` = 近似、`传说` = 低置信 — 见 [`SKILL_07a_人物生卒年推断.md`](SKILL_07a_人物生卒年推断.md)

### 模式 D：反思驱动的规则进化

每条规则在脚本注释或文档中**写明来源反思**（哪一轮、哪一类样本），便于回滚与维护。

- 实例：[`kg/events/reports/reflect_*.json`](../kg/events) 五轮事件年代反思 → 每轮 `corrections` 数组带 `confidence` 字段
- 实例：第三/四轮按章实体反思 → 沉淀进 [`skills/references/SKILL_03c1-v2-rules.md`](references/SKILL_03c1-v2-rules.md) 的 134 条规律

## 四、信号源类型（按强度递减）

| 强度 | 信号源 | 典型来源 |
|---|---|---|
| 极高 | 显式白/黑名单 | 手工词表、外部权威表（如《汉书·地理志》县表） |
| 高 | 注释/外部知识库交叉 | 三家注、年表、姓氏表、生卒年库 |
| 高 | 章节级上下文（题材已知） | 侯者年表 → 候选偏向"县"；本纪 → 偏向"职位" |
| 中 | 局部上下文窗口（前后 N 字） | 邻近实体、邻近动词、邻近虚词 |
| 中 | 全章/全书共现统计 | 同章节多次共现、全库高频搭配 |
| 低 | 后缀/字形启发式 | "山"结尾→山脉、"江/水"结尾→水域 |
| 低 | 并列传播 | 同列表/同句相邻项的多数类 |

**融合规则**：高强度信号一旦命中即终结判断；中低强度信号必须达到 `min_hits` 且 `majority` 才采纳；任何一层失败都向下兜底，最终未命中则**显式弃权**而非乱猜。

## 五、案例：地名地段分类（三阶段实例）

这是一个完整跑通「无中生有 → 反思纠正 → 置信度函数」三阶段的样板，可直接对照学习。

**完整设计文档**：[`doc/entities/地名反思/置信度函数设计.md`](../doc/entities/地名反思/置信度函数设计.md) —— 包含证据分层、权重锚点、聚合方法（MAX）、已知局限、R1-R7 演进历史等完整细节。

### 阶段 1 · 无中生有：从零产出标注与 baseline

[`SKILL_03h_地名地段分类.md`](SKILL_03h_地名地段分类.md) 定义了 14 类地段分类体系，并写下 L1-L5 分层启发式（白名单 → 章节上下文 → 共现 → 注释 → 并列传播）。脚本 [`classify_places.py`](../kg/entities/scripts/classify_places.py) 据此跑出 baseline，给 2116 个 place 实体打初版标签 ——**没有任何打分，只有命中/未命中**。

### 阶段 2 · 人工交互式纠正：从样本里反思

[`doc/entities/地名反思/第一轮_反思报告.md`](../doc/entities/地名反思/第一轮_反思报告.md) 对 462 条未分类项**逐条**判定，每 10 条做一次小反思，产出 `R1`-`Rn` 规则候选（如 R2「后缀 `浦`/`澨` → 水域」、R5「`X郊` 前接祭/祠/立祠 → 建筑」）。这些候选回填进 [`SKILL_03h`](SKILL_03h_地名地段分类.md) 与 `classify_places.py`，扩充白名单与共现模式。

**关键动作**：每次小反思只看 10 条，规则提炼即时回灌；不积压、不批量。

### 阶段 3 · 产出置信度函数：从证据强度归纳权重

[`doc/entities/地名反思/第二轮_反思报告.md`](../doc/entities/地名反思/第二轮_反思报告.md) 引入 [`confidence_report.py`](../kg/entities/scripts/confidence_report.py)，为每条 (name, category) 算最强证据得分。**权重表是从前两阶段经验里直接读出来的**：

- 人工验证白名单（KNOWN_COMMANDERIES）→ 1.00
- 显式 EXPLICIT_* / 汉书地理志县表 → 0.85-0.95
- 强后缀（山/水/郡/关）→ 0.80
- L3 共现 ≥3 次 → 0.65；2 次 → 0.55；1 次 → 0.40
- peer 传播 → 0.50；仅间接证据 → 0.30

**核心思想**：推理链条越长 → 单一规则置信度越低 → 多源交叉才能凑出高分。

第二轮内部又跑了一次**置信度驱动的迭代**：
1. 按报告挑出 264 条 `<0.50` 的低置信条目
2. 针对热点（单字地名、弱后缀、L3 单次命中）追加白名单与规则
3. 重跑 → 低置信条目 264 → **135**（降 49%）

**这就是反思归纳的闭环**：baseline → 反思 → 规则 → 置信度 → 低置信清单 → 反思 → 规则…… 每一轮都把模糊地带向白名单和强证据搬迁。

## 七、设计 Checklist

新设计或修订置信度函数前，逐项确认：

- [ ] 我手上是否有至少一轮**反思样本**？没有就先去做反思，不要凭空写公式
- [ ] 每个信号源是否能**指回**反思中具体的某类证据？无来源的信号删掉
- [ ] 信号是否按**强度分层**？强信号是否独占决定权？
- [ ] 关键阈值（`majority`、`min_hits`、窗口宽度）是否在脚本/文档中**注明回测样本量**？
- [ ] 低置信样本是**显式弃权 / 多标签**，而不是被强行归类？
- [ ] 输出格式是否便于下一轮反思**抽样**（如带候选层级、带命中信号）？
- [ ] 规则修改是否在 commit message 或反思工件中**留痕**，可回滚？

## 八、反模式

- ❌ 先写浮点加权公式 `0.4·A + 0.3·B + ...`，再去校验数据 → 调不动、不可解释
- ❌ 用单一阈值切 0-1 浮点 → 等价于赌博，不如三级标记
- ❌ 弱规则覆盖强规则的判断 → 永远先白名单/外部库优先
- ❌ 低置信样本**强行归类**而非弃权 → 污染下游、丢失反思线索
- ❌ 信号源没有反思来源、阈值没有回测来源 → 不可维护，下一轮没法演化
