# SKILL 01f: 繁简词库构造分析

> **核心价值**：通过失败案例和成功经验，总结出繁简词库构造的核心原则和方法论边界。

---

## 一、问题定义

### 1.1 目标

为OpenCC转换构建自定义词库（`custom-variants.json`），修正**上下文相关**的繁简转换错误。

**典型问题**：
- `"后"` → OpenCC统一转为`"後"`，但爵位应保持`"后"`
  - ✅ `"吕后"` → `"呂后"`（皇后，保持"后"）
  - ✅ `"后世"` → `"後世"`（时间，转为"後"）

### 1.2 核心挑战

如何区分以下三类差异：

| 差异类型 | 示例 | 是否应加入词库 |
|---------|------|--------------|
| **繁简转换差异** | `"后世"` OpenCC错转为`"后世"` | ✅ 应加入 |
| **异体字选择差异** | `"击"` OpenCC转`"擊"`，维基用`"撃"` | ❌ 不应加入 |
| **文本版本差异** | `"九年"` vs `"一年"` | ❌ 不应加入 |

## 二、三次尝试与经验

### 2.1 尝试1：自动逐字比对（失败）

#### 方法

```python
# 伪代码
for chapter in chapters:
    opencc_result = opencc.convert(simplified_text)
    wikisource_text = fetch_wikisource(chapter)
    diff = char_diff(opencc_result, wikisource_text)
    variants.append(diff)
```

#### 失败原因

提取到的8537条"差异"中，**99%是文本版本差异**：

```
差异示例：
"九" → "一"         # 年份不同
"君" → "臣"         # 人物称谓不同
"盲，而舜母死" → "更娶妻而生象"  # 整句不同
```

#### 教训

**不同版本的古籍不能直接逐字比对来提取繁简规则**。

维基文库底本与项目底本（中华书局《史记集解三家注索隐正义》）**不是同一版本**，提取到的是**校勘差异**，不是繁简差异。

---

### 2.2 尝试2：手动规则构造（成功）

#### 方法

1. **识别已知问题**：分析OpenCC转换错误（如"后"字问题）
2. **列举常见词汇**：基于《史记》文本特点，列举高频词
3. **构建精简规则集**：只添加确认必要的规则

#### 成果

**32条精准规则**（[`docs/data/custom-variants.json`](../docs/data/custom-variants.json)）：

```json
{
  "吕后": "呂后",    // 爵位，保持"后"
  "太后": "太后",
  "皇后": "皇后",
  "后世": "後世",    // 时间，转为"後"
  "其后": "其後",
  "即": "卽",        // 古籍常用"卽"
  "原": "願",        // "原"作"愿"时
  "于": "於",        // 介词
  "琅": "瑯"         // 地名
  // ... 共32条
}
```

**规则分类**：
- **"后"字上下文**（23条）：爵位用"后"，时间用"後"
- **其他异体字**（9条）：古籍专用字形

#### 优点

- ✅ **精准**：每条规则都解决实际问题
- ✅ **可维护**：规则少，易理解
- ✅ **可验证**：每条规则在《史记》中有实例

#### 缺点

- ⚠️ **覆盖不全**：可能遗漏部分异体字

### 2.3 尝试3：模糊匹配提取（失败）

#### 动机

希望通过改进算法，**自动**从维基文库提取更多规则，补充手动词库的覆盖不全问题。

#### 方法

```python
from difflib import SequenceMatcher

def extract_variants_fuzzy():
    for chapter in chapters:
        opencc_text = opencc.convert(simplified)
        wiki_text = fetch_wikisource(chapter)

        # 模糊对齐
        matcher = SequenceMatcher(None, opencc_text, wiki_text)
        for tag, i1, i2, j1, j2 in matcher.get_opcodes():
            if tag == 'replace':
                diff_pair = (opencc_text[i1:i2], wiki_text[j1:j2])

                # 过滤版本差异
                if is_likely_variant_not_version(diff_pair):
                    variants[diff_pair] += 1

    # 频率过滤（≥3次）
    return {k: v for k, v in variants.items() if freq >= 3}
```

**过滤规则**：
1. **长度过滤**：长度差异>2视为版本差异
2. **数字过滤**：纯数字视为年份差异
3. **已知异体字表验证**：80+已知繁简对
4. **频率统计**：≥3次出现才保留

#### 结果

提取**405条规则**，筛选出**48条高频规则**（≥3次）。

#### 验证结果（关键发现）

对48条高频规则进行OpenCC验证：

| 简体 | OpenCC结果 | 维基文库 | 类型 | 是否应添加 |
|-----|-----------|---------|------|----------|
| 击吴 | **擊**吳 | **撃**呉 | 异体字差异 | ❌ 否 |
| 录录 | **彔彔** | **錄錄** | 异体字差异 | ❌ 否 |
| 昆仑 | **崑**崙 | **昆**侖 | 异体字差异 | ❌ 否 |
| 谥为 | **諡**爲 | **謚**為 | 异体字差异 | ❌ 否 |
| 历 | **歷** | **曆** | 异体字差异 | ❌ 否 |
| "曰：" | "曰：" | **「**曰：**「** | 标点风格 | ❌ 否 |

**结论**：
- **23条引号规则**：标点风格差异（大陆`""`vs台湾`「」`）
- **22条异体字规则**：OpenCC已正确转换，只是异体字选择不同
- **3条重复规则**：与手动词库重复
- **0条有效新规则**

#### 失败原因

**无法区分"繁简转换差异"和"异体字选择差异"**：

1. **OpenCC已经正确完成了繁简转换**
   - `"击"` → OpenCC转为繁体`"擊"` ✅
   - `"录"` → OpenCC转为繁体`"彔"` ✅

2. **差异来源于异体字选择**
   - 维基文库选择异体字`"撃"`而非`"擊"`
   - 维基文库选择`"錄"`而非`"彔"`
   - 这是**编辑偏好**，不是OpenCC缺陷

3. **维基文库不适合作为标准**
   - 维基使用的繁体字是**特定异体字版本**
   - 与OpenCC的异体字选择不同（如`"擊"`vs`"撃"`）
   - 这些差异**不应该**作为我们的转换规则

#### 教训

1. **与维基文库比较不适合构建繁简词库**
2. **异体字选择差异 ≠ OpenCC缺陷**
3. **自动提取方法需要先验证OpenCC是否已正确处理**

#### 成功之处

虽然未提取到新规则，但：
- ✅ **验证了手动规则的高频性**：`"原"→"願"`出现63次，`"琅"→"瑯"`出现12次
- ✅ **模糊匹配成功过滤了版本差异和校勘差异**（相比尝试1的8537条噪声）
- ✅ **识别出了标点风格差异问题**（23条引号规则）

## 三、核心认识

### 3.1 三类差异的本质区别

| 差异类型 | 定义 | 示例 | OpenCC是否正确 | 是否加入词库 |
|---------|------|------|--------------|------------|
| **繁简转换差异** | OpenCC转换结果不符合语境 | `"后世"` 应转`"後世"`，OpenCC错转为`"后世"` | ❌ 错误 | ✅ 应加入 |
| **异体字选择差异** | OpenCC转换正确，但异体字选择不同 | `"击"` → OpenCC用`"擊"`，维基用`"撃"` | ✅ 正确 | ❌ 不加入 |
| **文本版本差异** | 底本不同导致的文字差异 | `"九年"` vs `"一年"` | - | ❌ 不加入 |

### 3.2 词库构造原则（修订）

1. ✅ **只添加OpenCC未正确转换的规则**
   - 必须先验证：`opencc.convert(简体) ≠ 期望繁体`

2. ❌ **不追求与维基文库完全一致**
   - 异体字选择是编辑偏好，不是转换问题

3. ✅ **聚焦核心问题**
   - 上下文相关的字符转换（如"后"字）
   - 古籍常用异体字（如"卽""願"）

4. ✅ **精简优先**
   - 32条已足够解决核心问题
   - 不追求"完整"词库

### 3.3 方法论边界

| 方法 | 适用场景 | 不适用场景 |
|-----|---------|-----------|
| **逐字比对** | 同一版本的繁简对照 | ❌ 不同版本的古籍 |
| **模糊匹配** | 过滤版本差异、识别趋势 | ❌ 区分转换差异和异体字差异 |
| **手动构建** | ✅ 解决已知核心问题 | 大规模覆盖（成本高） |
| **与维基比较** | 发现潜在问题 | ❌ 直接作为词库来源 |

## 四、工作流程（推荐）

### 4.1 构建新词库

```
1. 发现问题
   ↓
   在繁体显示中发现转换错误
   例：发现"吕后"显示为"吕後"

2. 确认类型
   ↓
   判断是OpenCC问题还是上下文问题
   测试：opencc.convert("吕后") → "吕後" ❌

3. 验证必要性
   ↓
   确认该词在《史记》中多次出现
   grep "吕后" archive/*.txt → 26处

4. 添加规则
   ↓
   在custom-variants.json中添加
   {"吕后": "呂后"}

5. 测试验证
   ↓
   在实际HTML中测试繁体显示
```

### 4.2 验证现有规则

```python
# 验证custom-variants.json中的规则是否必要
from opencc import OpenCC

cc = OpenCC('s2t')
with open('docs/data/custom-variants.json') as f:
    variants = json.load(f)

for simp, trad in variants.items():
    opencc_result = cc.convert(simp)
    if opencc_result == trad:
        print(f"⚠️ 规则冗余：{simp} → {trad}")
        print(f"   OpenCC已正确转换")
    else:
        print(f"✅ 规则必要：{simp}")
        print(f"   OpenCC: {opencc_result} ≠ 期望: {trad}")
```

### 4.3 评估自动提取结果

```python
# 评估自动提取的规则是否应该添加
def should_add_rule(simp, trad):
    # 1. 先检查OpenCC是否已正确处理
    opencc_result = cc.convert(simp)
    if opencc_result == trad:
        return False, "OpenCC已正确转换，这是异体字差异"

    # 2. 检查是否标点风格差异
    if is_punctuation_style_diff(simp, trad):
        return False, "这是标点风格差异，应独立处理"

    # 3. 检查频率
    freq = count_in_shiji(simp)
    if freq < 3:
        return False, f"频率过低（{freq}次）"

    # 4. 人工确认
    return True, "建议人工确认后添加"
```

## 七、经验总结

### 7.1 失败的尝试

| 尝试 | 方法 | 提取规则数 | 失败原因 |
|-----|------|-----------|---------|
| 尝试1 | 逐字比对 | 8537条 | 提取到文本校勘差异 |
| 尝试3 | 模糊匹配 | 48条（高频） | 提取到异体字选择差异 |

### 7.2 成功的做法

| 尝试 | 方法 | 规则数 | 成功原因 |
|-----|------|-------|---------|
| 尝试2 | 手动构建 | 32条 | 聚焦核心问题，精准解决 |

### 7.3 三个关键认识

1. **文本版本差异** ≠ 繁简转换差异（尝试1的教训）
2. **异体字选择差异** ≠ OpenCC缺陷（尝试3的教训）
3. **维基文库不适合作为繁简词库的构建来源**

### 7.4 方法论启示

对于古籍繁简词库构建：
- ✅ **以问题为导向**：发现实际转换错误再添加规则
- ✅ **精简优先**：32条精准规则 > 数百条含噪声的规则
- ✅ **验证驱动**：每条规则都需要OpenCC验证
- ❌ **避免自动化陷阱**：自动提取需要非常谨慎的验证
